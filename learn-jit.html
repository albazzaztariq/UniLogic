JIT Compilation

What it is, why it exists, and how it makes interpreters fast.

The Speed Problem

An interpreter reads each bytecode instruction, decides what to do with it, and does it. That decision step — called dispatch — happens for every single instruction your program executes. If your loop runs a million times, the interpreter makes a million dispatch decisions for every instruction in that loop. The actual work (adding two numbers, comparing values) is fast. The overhead of figuring out which work to do, over and over, is what makes interpreters slow.

A traditional compiler solves this by translating your entire program to native machine code before it ever runs. The CPU executes the instructions directly — no interpreter, no dispatch, no middleman. But the binary only works on the platform you compiled it for. A program compiled for x86 on Windows won't run on an ARM chip in a phone.

JIT compilation — Just-In-Time compilation — is the compromise. Start by interpreting. Watch which parts of the code run frequently. Compile just those parts to native machine code while the program is still running. The program starts immediately (no waiting for a full compilation pass), and the hot parts gradually get faster as the JIT compiles them. You get the portability of bytecode and, for the code that matters most, the speed of native execution.

How It Actually Works

Step 1: Profile. The interpreter runs your bytecode normally, but every basic block (a straight-line chunk of code with no branches) has a counter. Each time that block executes, the counter ticks up. This costs almost nothing — one increment per block — but it tells the runtime exactly where your program spends its time.

Step 2: Trigger. When a function's counter crosses a threshold (say, 10,000 executions), that function is considered "hot." The JIT compiler kicks in. The threshold is a tuning parameter — too low and you waste time compiling code that only runs briefly; too high and you leave performance on the table.

Step 3: Compile. The JIT takes the hot function's bytecode and compiles it to native machine code — real x86 or ARM instructions that the CPU executes directly. This compilation happens on a background thread so it doesn't block your program. Once the native code is ready, the runtime installs it.

Step 4: Execute. The next time that function is called, the interpreter doesn't interpret it. Instead, it jumps directly to the native code. Everything else that hasn't hit the threshold continues interpreting normally. Over time, the hottest parts of your program are all running as native machine code, and the cold parts (setup, initialization, error handlers) stay interpreted — which is fine, because they barely run.

Deoptimization — The Safety Net

To generate fast native code, the JIT makes assumptions. "This variable is always an integer." "This function is never redefined." "This branch is never taken." These assumptions let the JIT skip type checks, inline function calls, and eliminate dead code. The result is tight, fast machine code — as long as the assumptions hold.

But assumptions can break. A variable that was always an integer suddenly receives a string. A function gets monkey-patched. When this happens, the JIT can't just crash — it has to undo its work. It throws away the native code for that function and falls back to the interpreter, which handles every case correctly (just more slowly). This process is called deoptimization.

To make this possible, the JIT saves "checkpoints" at every assumption point — enough information to reconstruct the interpreter's state (which variables hold what values, where execution was) from the middle of the native code. When an assumption breaks, the runtime uses the checkpoint to resume interpreting from exactly the right place. It's like a trapeze act with a safety net: the JIT swings fast and high, but if something goes wrong, it falls safely back to the slower interpreter.

Did you know? JavaScript's V8 engine can deoptimize and reoptimize the same function hundreds of times during a single page load. Every time you pass a different type to a function, V8 might throw away the optimized version and start over. This is why writing "type-stable" JavaScript (always passing the same types to the same functions) makes your code measurably faster.

On-Stack Replacement — The Hard Part

Imagine your program's main() function contains a loop that runs 10 million times. After iteration 10,000 the JIT compiles main() to native code. But main() is already running — the interpreter is in the middle of it. You can't restart main() from the beginning just to use the native version. The next call to main() would use the native code, but there is no next call — it's the main function.

On-Stack Replacement (OSR) solves this. At the top of the loop, the interpreter checks: is JIT-compiled code available for this function? If yes, it transfers execution from the interpreter to the native code mid-function. It maps the interpreter's variables and state to CPU registers, and continues in native code from exactly where the interpreter left off. The program never restarts; it just silently switches engines.

Without OSR, the JIT is useless for long-running single-function programs — which is most programs that have a main loop. OSR is what makes JIT compilation practical for real-world code, and it's one of the hardest things to implement correctly because the interpreter state and the native code state have to match precisely at the transfer point.

Tiered Compilation

Not all JIT compilation is equal. A fast-but-simple JIT can compile quickly but produces mediocre native code (no inlining, no register optimization, no dead code elimination). A slow-but-thorough JIT produces excellent native code but takes longer to compile — time your program spends waiting.

The solution is tiers. Tier 1 (baseline): compile quickly with minimal optimization. The code runs faster than the interpreter but not as fast as it could. Good enough for warm code. Tier 2 (optimizing): if the code is really hot, recompile it with full optimization — inlining, dead code elimination, loop unrolling, sophisticated register allocation. Takes longer to compile, but the resulting code runs much faster. The baseline JIT buys you time while the optimizing JIT does its work.

Real systems use this approach. Java HotSpot has two tiers: C1 (baseline, compiles fast) and C2 (optimizing, compiles slow but produces fast code). Chrome's V8 engine has three tiers: Ignition (interpreter) to Sparkplug (baseline JIT) to TurboFan (optimizing JIT). UBI's specification defines a two-tier approach: an interpreter with quickening (adaptive specialization) as the first tier, and an optimizing JIT as the second.

How UBI Uses All of This

UBI combines these techniques into a two-tier execution pipeline. The interpreter runs bytecode with adaptive specialization — generic opcodes like ADD rewrite themselves to ADD_INT after observing integer operands, skipping type dispatch on all subsequent executions. Hotness counters in the bytecode track which blocks run frequently. When a block crosses the compilation threshold, the JIT compiles that function to native machine code using the profiling data the interpreter has already collected.

The register-based architecture means fewer instructions per operation (one ADD r2, r0, r1 instead of four stack operations), and the metadata the compiler embeds in the .ubi file — control flow edges, kill bits, block boundaries — gives the JIT a pre-computed control flow graph and register liveness information. No rediscovery needed. The JIT starts with what the compiler already knew.

Deoptimization checkpoints at every guard point ensure correctness: if a type assumption breaks, execution falls back to the interpreter seamlessly. On-stack replacement handles the common case of long-running loops — the JIT can take over mid-function without waiting for the next call. The result is a system where cold code runs immediately (no upfront compilation delay), warm code self-optimizes through quickening, and hot code compiles to native machine instructions.

Did you know? The term "Just-In-Time" was borrowed from manufacturing. Toyota's JIT manufacturing system (1970s) produces parts exactly when they're needed, not in advance. JIT compilation does the same thing: compile code exactly when it's needed (when it's hot), not in advance (at program start). Both systems avoid wasting resources on work that might never be used.
