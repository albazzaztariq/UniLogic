Features

Every language makes decisions for you. UL lets you make them yourself.

Runtime Behavior
In Python, Java, C, and JavaScript, the runtime behavior of your program is fixed by whichever language you chose. GC or no GC. Safe or unsafe. You don't get to decide.
UL declares runtime behavior per file. Memory model, safety level, integer width, concurrency model — each file sets its own policy. The compiler enforces that files with conflicting policies don't call each other without a warning.
Memory Model
Python gives you GC and nothing else. C gives you manual memory and nothing else. Rust gives you the borrow checker and nothing else. You pick a language, you pick a memory model, forever.
UL has four memory models — gc, manual, refcount, arena — selectable per file. Write the hot path with manual memory for deterministic performance. Write the rest with gc. The compiler tells you when those two worlds try to touch each other unsafely.
Compile Targets
Every language compiles to one thing. Python runs on the Python interpreter. Java runs on the JVM. C compiles to one platform's native binary. Getting your code to run somewhere new means rewriting it.
One UL source file compiles to C, Python, JavaScript, LLVM IR, and WebAssembly. Same code. No changes. The target is a build flag, not a language choice.
Integer Precision
JavaScript has no integer type — everything is a 64-bit float, and integers above 2³³ silently lose precision. Python integers are arbitrary precision, which sounds safe until you hit a C library boundary and something truncates without warning.
UL declares integer width as a DR setting — 32, 64, or platform. With safety = checked, overflow is caught. The compiler warns when mismatched widths cross file boundaries.
Error Handling
C returns error codes you can silently ignore. Python throws exceptions that unwind arbitrarily. Go returns errors you must check manually every line. None of these are enforced by the compiler.
UL has Result types built in. Functions return ok value or error message. The ? operator propagates errors up the call chain automatically. At the boundary of a non-result function, errors print to stderr and exit. The compiler catches unhandled paths.
Type System
Python has no static types unless you add mypy, which isn't enforced at runtime. JavaScript has no types unless you add TypeScript, which compiles away entirely. C has types but no way to make them configurable.
UL is statically typed by default. If you want Python-style dynamic flexibility, set @dr types = dynamic. The compiler enforces whichever mode you're in. You choose per file.
Profiling
Profiling in Python requires cProfile, line_profiler, memory_profiler — separate tools, separate install, separate report. Getting timing, memory, and call graph data simultaneously means three different tools with three different output formats that don't talk to each other.
UL's profiler is one flag: --profile. Compile with it, run your program, open VS Code. Call timers, memory tracker, flame graph, DR behavior monitor — all in one panel reading one log file.
Cross-File Policy Conflicts
No language detects when a function in a garbage-collected file calls a function in a manually-managed file and the callee frees memory the caller still holds. This is one of the most common causes of memory corruption in mixed codebases. It's invisible until it crashes.
UL's compiler reads the DR settings of every file in the workspace. When a call crosses a memory model boundary, it emits a warning at the call site — before compilation, before execution, before the crash.
Deployment
Python requires the interpreter installed on the target machine plus all dependencies at the right versions. JavaScript requires a runtime. Java requires the JVM. Getting your code to run on a machine you don't control involves package managers, version conflicts, and environment setup.
UL compiles to a native binary that runs standalone. Or compile to WASM and run in any browser. Or run on the UL VM for portability without native compilation. The deployment model is a choice, not a constraint.
AI Workloads
PyTorch is written in Python — the single worst language for performance-critical computation. Mixed precision requires .half() calls scattered everywhere. Quantization is a post-training library operation, not a type. Kernel fusion happens if torch.compile decides to do it. None of it is under your control.
UL declares precision, quantization, sparsity, and vectorization as type-level and file-level settings. The compiler applies them. No library calls. No hoping the framework notices the pattern. Compile-time decisions with full information.
