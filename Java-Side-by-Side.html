Example 1 — HttpRequest
Java
class HttpRequest {
  private String url;
  private String method;
  private int timeout;
  private boolean followRedirects;
  private int retries;
  private String authToken;

  HttpRequest(String url, String method,
    int timeout, boolean followRedirects,
    int retries, String authToken) {
    this.url = url;
    this.method = method;
    this.timeout = timeout;
    this.followRedirects = followRedirects;
    this.retries = retries;
    this.authToken = authToken;
  }
}
UL
type HttpRequest
  string url, method
  int timeout, followRedirects, retries

end type

function new_request(string url, string method, int timeout = 30,
  bool followRedirects = true, int retries = 3, string authToken = empty)
  return HttpRequest(url, method, timeout, followRedirects, retries, authToken)
end function

Java forces every field to be written three times: once as a class member declaration, once as a constructor parameter, and once as a this.field = param assignment. For a six-field object that is eighteen lines of mechanical repetition before the class does anything.

UL separates the data layout from construction. The type block declares fields once. A constructor function then handles initialization with default parameter values — callers only pass what they need to override. Java has no native default parameters, so every constructor variant requires an overloaded method or a builder pattern.

The UL version carries the same information in roughly half the lines, with no loss of clarity.
