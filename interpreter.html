Bytecode Interpreter

UBI — the UniLogic bytecode interpreter

Python (stack-based) — x = a + b:

LOAD_FAST    0 (a)      -- push a onto operand stack
LOAD_FAST    1 (b)      -- push b onto operand stack
BINARY_ADD              -- pop two, add, push result
STORE_FAST   2 (x)      -- pop result, store in x

4 instructions, 6 stack operations (3 pushes, 3 pops). Every intermediate value goes through the operand stack in memory.

UBI (register-based) — same operation:

ADD r2, r0, r1    -- r2 = r0 + r1

1 instruction, 0 stack operations. The values stay in registers.

A register-based interpreter that executes compiled UniLogic bytecode directly.

UL source files (.ul) compile to UL bytecode (.ubi), then hand that file to UBI. It loads the instructions, sets up the register file, and runs. The XPile toolchain performs the compilation to bytecode.

What It Is

The UniLogic Bytecode Interpreter (UBI) runs .ubi files — compiled UniLogic bytecode. It sits alongside the existing transpilation targets (C, Python, JS, LLVM IR) as a self-contained execution path: instead of emitting source code and handing off to another compiler, you get a binary bytecode format that UBI runs directly.

The interpreter is register-based. It maintains a flat register file, a program counter, a call stack for function calls and returns, and a heap for dynamic allocation. Each instruction is 32 bits wide. The interpreter fetches, decodes, and dispatches one instruction per cycle in a straightforward switch-based dispatch loop.

This is useful when you want a portable execution environment without pulling in a C compiler or a Python runtime. Ship the .ubi file and the interpreter binary, and it runs.

Why Register-Based

Most teaching interpreters use a stack machine because it is easy to implement: push operands, execute an opcode, result stays on the stack. The JVM and CPython both work this way. UBI uses a register machine instead, and the reason is direct: fewer instructions.

Shi et al. (VEE 2005, "Virtual Machine Showdown: Stack Versus Registers") implemented the same programs on both architectures and found that register-based interpreters execute roughly 2–3× fewer instructions than equivalent stack interpreters. The trade-off is that each instruction is wider (you have to encode register operands), but that is exactly the deal UBI makes: fixed 32-bit instructions, fewer of them, less dispatch overhead.

RegCPython — a register-based reimplementation of CPython — confirmed the register architecture is appreciably faster and that "compilation speed is never a reason for choosing stack architecture" (Brito & Valente, ACM SAC 2023).

For a language like UniLogic where the type system is doing real work at compile time, it makes sense to lean into that and emit efficient bytecode rather than carrying the overhead of a stack machine at runtime.

Instruction Format

Every UBI instruction is exactly 32 bits. There are three encoding forms depending on what the instruction needs to express:

Form A — Register/Register (arithmetic, logic, comparisons)

bits: [31–24] opcode  |  [23–16] rd  |  [15–8] rs1  |  [7–0] rs2

opcode8 bits
rd8 bits
rs18 bits
rs28 bits

Form B — Register/Immediate (loads, branches, calls)

bits: [31–24] opcode  |  [23–16] rd  |  [15–0] immediate

opcode8 bits
rd8 bits
immediate16 bits

Form C — No operands (halt, nop, return)

bits: [31–24] opcode  |  [23–0] unused (zero)

opcode8 bits
unused24 bits (zero)

The fixed-width format simplifies the decoder: read 4 bytes, extract the top 8 bits to get the opcode, then branch on encoding form. No variable-length parsing, no prefix bytes.

Register naming

Registers are referenced by index in the bytecode. The register file size is configurable at interpreter startup. Calling convention and register assignment are handled by the compiler (XPile's UBI backend) — the interpreter itself just operates on indices.

Tiered ISA

The instruction set is split into tiers. This is partly pragmatic (build the basics first, validate them, then add higher-level ops) and partly architectural (a future JIT only needs to handle Tier 0 to be correct; Tier 1 is an optimization layer on top).

Tier 0 — Core

The fundamentals. Anything a correct program needs.

CategoryInstructions
Integer arithmeticADD, SUB, MUL, DIV, MOD
BitwiseAND, OR, XOR, NOT, SHL, SHR
ComparisonEQ, NEQ, LT, LTE, GT, GTE
Control flowJMP, JMPIF, JMPIFN, CALL, RET, HALT
Data movementLOAD, STORE, LOADI (immediate), MOV
MemoryALLOC, FREE (used by manual/gc modes)
I/OPRINT, PRINTLN, READ

Tier 1 — Semantic Operations

Higher-level ops that map directly to UniLogic language constructs. They could be lowered to Tier 0 sequences, but encoding them as single instructions keeps the bytecode compact and makes the interpreter's intent obvious.

InstructionWhat It Does
ASSERT_TYPERuntime type assertion — traps if the value in rd does not match the expected type tag
CASTExplicit type conversion between numeric types
BOUNDS_CHECKArray bounds check with a clean trap on failure rather than a silent bad access
REFCOUNT_INC / DECIncrement/decrement reference counts when running in refcount memory mode
GC_MARKMark an allocation as reachable during a GC cycle
ARENA_PUSH / POPPush and pop arena frames for scoped bulk allocation
Future tier Tier 2 is reserved for JIT-facing hints: loop annotations, hot-path markers, inline cache slots. Nothing there yet — the interpreter executes Tier 0 and Tier 1 today.

Memory Modes

UBI supports four memory management strategies, selected at compile time via the @mem annotation in UniLogic source. The interpreter adapts its allocation and free behavior based on which mode the bytecode was compiled for.

ModeHow it works
manualExplicit ALLOC and FREE instructions; the programmer controls every allocation lifetime, no runtime overhead.
gcA simple mark-and-sweep collector; GC_MARK instructions identify reachable objects, the collector sweeps the rest at configurable intervals.
refcountEach heap allocation carries a reference count; REFCOUNT_INC and REFCOUNT_DEC maintain it, and an allocation is freed when the count hits zero.
arenaAllocations are made from a contiguous region; ARENA_PUSH opens a frame and ARENA_POP frees everything in it at once — good for temporary scratch memory with a clear scope.

The mode is fixed per compilation unit. You do not switch memory modes at runtime.

Usage

Two steps: compile the .ul source to bytecode, then run it.

Step 1 — Compile to UBI bytecode

# -t ubi selects the UBI bytecode target
ul build test.ul -t ubi

This produces test.ubi in the current directory.

Step 2 — Run the bytecode

ULBytecode.exe test.ubi

The interpreter loads the .ubi file, validates the header, initializes the register file and heap according to the memory mode embedded in the bytecode, and starts executing from the entry point.

With a memory mode annotation

# In your .ul source, annotate the memory mode:
@mem(arena)
fn main() {
    -- your code here
}
# Then compile and run as normal:
ul build main.ul -t ubi
ULBytecode.exe main.ubi
Default memory mode If no @mem annotation is present, the compiler defaults to manual. You get explicit alloc/free and no runtime overhead from a collector.

Debug Stripping

By default, the compiler embeds a debug section in every .ubi file. This section contains variable names, type information, function signatures, and a line number table that maps bytecode offsets back to original source lines. This is what powers readable error messages, the decompiler (ubi2ul), and future debugger integration.

For distribution, you can strip the debug section entirely:

ul build app.ul -t ubi --strip

A stripped .ubi file runs identically — the interpreter does not need the debug section to execute. The difference is in what happens when something goes wrong or when someone tries to decompile it:

BehaviourWith debug sectionStripped
Error messagesFunction name, source line number, variable nameFunction index, instruction offset only
Decompilation (ubi2ul)Near-perfect source reconstruction with original namesGeneric slot names (_r0, _r1), flat control flow
File sizeLarger (debug tables + string heap)Smaller (code + constant pool only)

Debug section contents

The debug section is appended after the instruction blocks and marked by an 8-byte sentinel (UBDB + 0xDB00DB00). It contains four tables:

TableWhat it stores
Variable Name TableRegister slot → variable name mapping, per function. Scope start offsets for each variable.
Type TableStruct, object, enum, and array type definitions with field names and types.
Function Debug TableFunction index → name, parameter list with types, return type, first source line.
Line Number TableInstruction offset → source line mapping, delta-encoded with zigzag compression.

All entries reference a shared string heap at the start of the debug section. Integer values are ULEB128-encoded for compact storage.

When to strip Strip when shipping to end users or when the binary will be distributed publicly and you do not want source-level decompilation. Keep the debug section during development and testing — the line number table alone is worth the extra bytes.
