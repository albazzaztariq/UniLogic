XPile — Transpiler

Translation

XPile is UL's transpiler. It converts source code between UL and any supported language in both directions. Translation is not just mechanical substitution — it is configurable, so output code reads the way the target codebase expects it to.

XPile handles two directions: importing foreign code into UL so you can work in it, and exporting UL back to a target language for deployment. Both directions are lossless in behaviour. Style is a separate concern, controlled by translation profiles.

What is XPile

XPile is a standalone tool that ships with UniLogic. It reads source code in a supported language and produces equivalent source code in another. It is not a compiler — it does not produce binaries. It produces source that can then be compiled or run by whatever toolchain the target language uses.

Supported translation directions:

FromTo
CUL
PythonUL
JavaScriptUL
JavaUL
ULC, Python, JavaScript, Java

The compiler (pragma) handles the UL → target direction as part of its normal build pipeline. XPile is the dedicated tool for the reverse — and for round-tripping.

Round-trip workflow

The primary use case for XPile is working in a foreign codebase. You translate the foreign source into UL, do your work in UL, then translate back to the original language for deployment. The rest of the team never sees UL. They see their own language, in their own style.

their C
→
UL
→
your work
→
UL
→
C they deploy

The challenge in the last step is that the output C has to look like their C, not like C produced by a tool that has never seen their codebase before. Different teams use different idioms, different loop styles, different allocation functions, different naming conventions. If the output code looks foreign, it creates friction — and friction turns into resistance to the tool.

This is what translation profiles solve.

Translation profiles

A translation profile is a configuration file that defines how UL constructs map to syntax in a specific target language. Profiles are per-target — you write a profile for C, a separate one for Java, and so on. Profiles are optional: if you don't provide one, XPile uses its defaults.

A profile does not change what the code does. It only changes how it is written. The output with a custom profile and the output with the default profile must be behaviourally identical.

Profiles give you control over style. They do not give you control over semantics. XPile validates this — see Profile checker.

Profiles are stored as .xpile files and referenced by name when invoking the transpiler:

xpile input.run --to c --profile team-style.xpile

What is mappable

Every mappable construct has a set of defined equivalent forms. You can choose any of them, or write your own — with the caveat that custom forms are checked for equivalence and flagged if XPile cannot confirm they produce the same behaviour. The following constructs are mappable per target language.

Loops

UL Default C output Example custom mapping
for i in 0..5 for (int i = 0; i < 5; i++) for (i = 0; i < 5; i += 1)
while cond while (cond) while (cond)
loop for (;;) while (1)

Variable declarations

UL Default C output Example custom mapping
let x = 5 int x = 5; int x = 5;  // same
let x: int32 = 5 int32_t x = 5; int x = 5;  // if team uses plain int

Memory allocation

UL Default C output Mappable alternatives
alloc(T) malloc(sizeof(T)) calloc(1, sizeof(T))  ·  custom allocator call
free(ptr) free(ptr) custom free call

Null checks

UL Default C output Mappable alternatives
if x == null if (x == NULL) if (!x)  ·  if (x == 0)

Boolean literals

UL Default C output Mappable alternatives
true / false true / false (stdbool.h) 1 / 0  ·  TRUE / FALSE (custom macro)

Function signatures

UL Default C output Mappable alternatives
fn add(a: int, b: int) → int int add(int a, int b) int add(int a, int b)  // same — C has one form

Profile format

A profile is a plain text file. Each entry maps a UL construct to its target-language form. Entries use a simple construct: "template" syntax. Templates use {} placeholders for the parts XPile fills in.

// team-style.xpile — target: c

target: c

// Loop style
for_range: "for ({type} {var} = {start}; {var} < {end}; {var} += 1)"

// Allocation — team uses calloc instead of malloc
alloc: "calloc(1, sizeof({type}))"

// Null style — team prefers !ptr over ptr == NULL
null_check: "(!{expr})"

// Boolean literals — team uses 1/0, no stdbool
bool_true: "1"
bool_false: "0"

Any construct not listed in the profile uses the default. You only need to specify what you want to override.

Profile checker

XPile validates every profile before using it. Validation has two tiers that operate independently.

Error Syntax validity
The output produced by the mapping must be syntactically valid in the target language. XPile parses the rendered template and checks it against the target language's grammar. If the result is not valid syntax, the profile is rejected and translation does not proceed.
error: profile 'team-style.xpile' — construct 'for_range'
rendered output is not valid C:
for (int i = 0; i < 5 i++) // missing semicolon
translation aborted
Warning Behavioural equivalence
XPile checks whether the custom mapping produces the same behaviour as one of UL's defined equivalents for that construct. It does not execute or prove the code — it compares the mapping structurally against the known-equivalent forms. If the mapping does not match any of them, a warning is issued and translation continues.

This is not an error because the mapping may still be correct — XPile simply cannot confirm it. If you proceed past this warning, the behaviour of the output relative to the UL source is your responsibility.

warning: profile 'team-style.xpile' — construct 'for_range'
mapping does not match any defined equivalent
known equivalents: for(init;cond;incr) · while(cond){...incr}
your mapping: for(items=0 while items<5 do items+=items)
behavioural equivalence unconfirmed — output may be undefined
translation proceeded with warning
The warning tier exists because XPile defines a finite set of known-equivalent forms for each construct. If your mapping is outside that set, XPile has no basis to confirm it is equivalent. It may be — but you are outside checked territory. The warning is there to make that explicit, not to block you.
