Why This Matters

Every programming language makes decisions for you. How memory is managed, how types are checked, how threads work — these are baked into the language before you write a single line. Pick Python and it automatically cleans up your variables from memory whether you want it to or not. Pick C and you're responsible for every byte of memory you touch.

These aren't just academic details. They determine whether your program leaks memory, crashes under load, runs on a microcontroller, or works in a browser. Most developers never think about them because most languages never let you choose. UL takes these aspects of other languages and makes them features and options.

This tutorial explains what those choices are, what they mean, and why having control over them changes what you can build.

Section 2 of 15

What is Runtime

The Runtime Environment, or just "runtime" for short, is the system that supports your program while it's actually running. It handles things like memory management, type checking, and threading — everything that happens after the compiler is done and the program starts executing. Every language has one, whether you see it or not.

Runtime behavior is the set of rules that runtime follows. How does it allocate memory? Does it check array bounds? Does it use one thread or many? In most languages, these rules are fixed and you don't get a say. Python automatically cleans up unused memory. C makes you do it yourself. Java checks every array access. C skips the check.

Each choice has tradeoffs. The next few sections look at how four major languages handle this, and what you give up in each case.

Section 3 of 15

Runtime Concepts You Need to Know

Allocation — asking the system for a chunk of memory to store data in. Every variable, object, and array lives in memory that was allocated.

Freeing / deallocation — giving memory back when you're done with it.

Memory leak — when allocated memory is never freed. It stays reserved forever, unavailable to anything else, until the program ends.

Garbage collection (GC) — an automatic system that finds memory nothing is using anymore and frees it for you. The runtime decides when, not you. Automatic also means no control over timing or behavior.

Manual memory — you allocate and free memory yourself, explicitly, in your code. Faster execution comes with extra work and dangers.

Reference counting — each object has a counter tracking how many things point to it. When the counter hits zero, it frees itself immediately. Fails when two objects point to each other (a cycle).

Arena — one big block of memory reserved up front. Allocations just move a pointer forward. When the scope ends, the entire block is freed at once. Fast, but every pointer into it dies when the scope closes.

Section 4 of 15

Runtime Concepts (continued)

Threads — independent paths of execution running at the same time inside one program, potentially on multiple CPU cores. They let a program do multiple things simultaneously.

Concurrency — when a program manages multiple tasks at once. This can mean threads running in parallel, or a single thread switching between tasks quickly (like JavaScript's event loop).

GIL (Global Interpreter Lock) — a lock in Python that only lets one thread run Python code at a time, even on multi-core machines. It exists to protect Python's internal data structures, but it prevents true parallel execution of Python code.

Bounds checking — verifying that an array index is within the valid range before accessing it. Java does this automatically. C does not — writing past the end of an array corrupts whatever memory sits next to it.

Type checking — verifying that a value matches the type expected. Static type checking happens at compile time (Java, C). Dynamic type checking happens at runtime (Python, JavaScript) — errors only appear when that line of code actually executes.

Section 5 of 15

Python

Python manages memory for you using garbage collection (see previous page). You never call free(). Types are checked at runtime — you can put a string where an integer was expected and Python won't complain until that line actually executes.

This makes Python fast to write and forgiving to learn. It also means bugs hide until production, performance is 10–100x slower than compiled languages, and the GIL prevents true CPU parallelism — only one thread can run Python code at a time, even if your machine has 16 cores. You can't turn any of this off. There's no manual memory mode. There's no strict types switch.

For scripting and prototyping, these tradeoffs are fine. For systems programming, embedded work, or latency-sensitive applications, they're dealbreakers.

Section 6 of 15

C

C gives you total control and total responsibility. You allocate memory with malloc() and free it with free(). The compiler trusts everything you write — no bounds checking, no null-pointer guards, no type safety beyond what you declare. If you write past the end of an array, C will happily corrupt whatever sits next to it in memory.

This makes C the fastest language for systems work. Operating systems, databases, and embedded firmware are written in C because nothing is wasted on safety checks you didn't ask for. The cost: memory bugs are the #1 source of security vulnerabilities. Google and Microsoft both report that ~70% of their critical CVEs are memory safety issues in C and C++ code.

C's runtime behavior is fixed at the opposite extreme from Python — manual everything, checked nothing. You can't ask for automatic memory cleanup. You can't ask for bounds checking. The language doesn't offer those options.

Section 7 of 15

JavaScript

JavaScript runs in a browser engine (V8, SpiderMonkey) that automatically cleans up unused memory, uses dynamic types, and runs on a single-threaded event loop. You can't manage memory directly — there's no pointer, no allocator, no way to control when objects are freed. The engine decides.

The event loop means JavaScript handles concurrency through callbacks and promises rather than threads. This avoids data races entirely but makes CPU-heavy work block the entire UI. Web Workers exist but run in isolated contexts with message-passing — no shared memory by default.

Like Python, JavaScript's runtime is take-it-or-leave-it. You get automatic memory cleanup, dynamic types, and single-threaded execution. There's no switch for manual memory, strict types, or real threads. The language was designed for the browser and its runtime reflects that.

Section 8 of 15

Java

Java runs on the JVM with automatic memory cleanup, strict static types, and built-in threading. You create objects with new and never think about freeing them — the runtime detects when they're no longer used and reclaims the memory. The JVM's JIT compiler optimizes hot paths to near-native speed over time.

Java's automatic cleanup system is sophisticated (multiple implementations exist) but still pauses threads to scan and free memory. For most applications this is invisible. For low-latency trading systems or real-time audio, pauses of even a few milliseconds are unacceptable. You can tune the cleanup behavior but you can't turn it off.

The JVM gives you real threads and a strong type system. What it doesn't give you is a choice. Automatic cleanup is always on. Bounds checking is always on. The type system is always strict. These are good defaults, but they're not configurable.

Section 9 of 15

The DR System

UniLogic's Dynamic Runtime lets you declare runtime behavior per file. Instead of inheriting a fixed set of rules from the language, you state what you want at the top of each source file: memory: gc or memory: manual. safety: checked or safety: unchecked. concurrency: threaded or concurrency: async.

The compiler reads these directives and enforces them. If two files with incompatible settings call each other, the compiler flags it before you run anything. A file using automatic cleanup passing a pointer to a manual-memory file gets a warning at the call site — not a crash at 3am.

Memory: Garbage Collection

Garbage collection is a system that automatically frees memory you're no longer using. Instead of you deciding when to clean up, the runtime periodically scans for objects that nothing points to anymore and deletes them for you. The catch: to do this safely, it has to pause everything else your program is doing while it scans. This is called "stop-the-world." Press Play to see the cycle in action.

Memory: Manual

Manual memory means there is no automatic cleanup. You explicitly ask for memory when you need it (memtake) and explicitly give it back when you're done (memgive). Nothing happens automatically — if you forget to give it back, it's gone forever. The animation walks through correct usage, then shows two classic bugs: a leak and a use-after-free.

Section 12 of 15

Memory: Reference Counting

Reference counting means each object tracks how many things point to it. When the count hits zero, it frees itself immediately — no pause needed. The tradeoff: if two objects point to each other, neither count ever hits zero and neither ever frees. This is called a reference cycle, and it's a permanent memory leak that counting alone cannot fix.

Section 13 of 15

Memory: Arena

An arena pre-allocates one large block of memory. Every allocation just advances a pointer — no searching, no bookkeeping. When the scope ends, the entire block is freed at once. Extremely fast and zero fragmentation. The risk: any pointer into the arena becomes invalid the instant the scope closes.
