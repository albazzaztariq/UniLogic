Bytecode Interpreters

What they are, how they evolved, and why the design decisions matter.

What Is Bytecode

When you compile a C program, the compiler translates your source code into machine instructions that the CPU executes directly. The binary is specific to your CPU and operating system. Move it to a different machine and it may not run.

Bytecode is the middle ground. Instead of compiling to machine code, the compiler translates your source into a set of instructions for a software processor — an interpreter. The interpreter reads these instructions one at a time and executes them. The bytecode file is portable (it runs wherever the interpreter runs), but slower than native code because there's a layer between your program and the hardware.

Source code  →  Compiler  →  Bytecode (.class, .pyc, .ubi)
                                      ↓
                               Interpreter reads & executes
                                      ↓
                                   CPU runs the interpreter

The tradeoff: you lose some speed but gain portability. The question every bytecode system has to answer is: how much speed can you claw back?

How We Got Here

1966
O-code (BCPL) — Martin Richards designed one of the first portable intermediate codes. BCPL compiled to O-code, which a small interpreter executed. The idea: write the compiler once, port only the interpreter.
1978
p-code (UCSD Pascal) — The UCSD Pascal system compiled to p-code (pseudo-code), interpreted by a p-machine. This let Pascal run on Apple II, Z80, PDP-11 — wildly different hardware — from the same bytecode. It proved the model worked for real software.
1980
Smalltalk-80 — Xerox PARC's Smalltalk compiled to bytecode for a stack-based virtual machine. This is where many modern interpreter ideas originated: garbage collection integrated with the runtime, bytecoded message dispatch, and the idea that the runtime could observe program behavior and optimize accordingly.
1987
Self — David Ungar and Randall Smith at Stanford (later Sun Microsystems Labs). Self looked like Smalltalk but ran dramatically faster because of three inventions that would reshape the entire field: adaptive optimization (observe what the program actually does, optimize for that), polymorphic inline caches (remember what types each call site has seen), and dynamic deoptimization (undo optimizations when assumptions break). Nearly every technique in modern interpreters traces back to Self.
1995
Java Virtual Machine — See below.
1993
Lua — Roberto Ierusalimschy and team at PUC-Rio. Started stack-based, switched to register-based in Lua 5.0 (2003). Mike Pall's LuaJIT (2005) became one of the fastest dynamic language runtimes ever built — a tracing JIT that compiles hot loops to native code at runtime. LuaJIT's NaN-boxing value representation (packing types into unused IEEE 754 bit patterns) influenced nearly every interpreter designed after it.
2007
Android Dalvik → ART — Google's Dalvik VM ran Java bytecode on phones. In 2013 they replaced it with ART (Android Runtime), which compiles bytecode ahead-of-time to native code at install. ART's bytecode format carries compiler-provided metadata — a design philosophy UBI shares.

The JVM — 30 Years and Still Running

In 1995, James Gosling and the team at Sun Microsystems shipped the Java Virtual Machine. Sun was a computing powerhouse — the company that built SPARC processors, Solaris, NFS, and employed many of the researchers who created Self. When they designed the JVM, they had access to some of the best systems minds in the industry.

Their design philosophy was pragmatic: keep the bytecode format simple. Put the intelligence in the runtime.

The JVM's .class file format is stack-based and deliberately minimal. No optimization hints, no control flow metadata, no register allocation. The compiler's job is just to produce correct bytecode. The runtime (HotSpot, shipped 1999) handles everything else: profiling which code is hot, compiling it to native code, inlining methods, eliminating dead code, deoptimizing when assumptions break.

This separation turned out to be brilliant. The .class format hasn't fundamentally changed since 1995, but the runtime has been rewritten multiple times, getting faster with each generation. And because the format is stable, languages invented decades later can target it:

Languages running on the JVM today Kotlin (2011, now Google's preferred language for Android), Scala (2004, functional + object-oriented), Clojure (2007, a Lisp), Groovy (2003, dynamic scripting), JRuby, Jython — all compiling to the same bytecode format designed in 1995. The JVM outlived the company that created it. Sun was acquired by Oracle in 2010, but the machine keeps running.

The key people from Self — Urs Hölzle, Lars Bak, Craig Chambers — went on to build HotSpot (the JVM's JIT compiler), and later V8 (Chrome's JavaScript engine) and Dart. The techniques they invented for Self in 1987 are still the foundation of every fast interpreter and JIT compiler in production today.

CPython — The Irony of the World's Most Popular Language

Python is everywhere. It's the language of machine learning, data science, web backends, scripting, automation. And its default interpreter, CPython, is one of the slowest bytecode runtimes in widespread use.

CPython is stack-based. Every a + b is three instructions: push a, push b, add. Every operation dispatches through a central switch statement with poor branch prediction. There's a Global Interpreter Lock (GIL) that prevents true multithreading. Until version 3.13 (2024), there was no JIT compiler at all.

The numbers are stark. PyPy — an alternative Python interpreter with a tracing JIT — is roughly 18x faster than CPython 3.14. Node.js (V8) is about 6x faster. CPython is closer to a shell script in performance than to a compiled language.

The irony: Python is the dominant language for AI and ML, which are among the most performance-sensitive applications in computing. The resolution is that Python in ML is mostly orchestration — it calls into C++ and CUDA libraries (NumPy, PyTorch, TensorFlow) that do the actual computation. Python is the glue, not the engine. But even as glue, the overhead adds up: data preprocessing, model evaluation loops, hyperparameter sweeps — all running at CPython speed.

CPython's escape hatches Cython compiles Python-like syntax to C, then GCC compiles to native. PyPy replaces the interpreter with a tracing JIT. Numba JIT-compiles numeric Python functions via LLVM. Each solves part of the problem but fragments the ecosystem — code that works in CPython may not work in PyPy, and Cython is a different language with different syntax. There's no single Python that's both compatible and fast.

What UBI Does Differently

UBI starts from the same research lineage — Self, HotSpot, LuaJIT, V8 — but with 30 years of hindsight. The design makes three deliberate choices that diverge from the JVM model:

1. Register-based, not stack-based

Where a + b takes 4 instructions in CPython (push, push, add, store), UBI does it in 1: ADD r2, r0, r1. Research shows register-based interpreters execute 2–3x fewer instructions for the same program. Fewer instructions means fewer trips through the dispatch loop, which is where interpreters spend most of their time.

2. Intelligence in the bytecode, not just the runtime

The JVM kept its bytecode simple and put all the smarts in HotSpot. UBI takes the modern approach (shared with Android's ART and WebAssembly): the compiler encodes metadata into the bytecode that the runtime can use immediately. Block hotness counters, control flow edges, kill bits for register reuse, tiered opcodes — all computed at compile time and shipped in the .ubi file. The interpreter doesn't have to rediscover what the compiler already knew.

3. Adaptive specialization from day one

CPython retrofitted type specialization in version 3.11 (PEP 659), decades after the interpreter shipped. UBI was designed with quickening built in: generic opcodes rewrite themselves to type-specialized variants after the first execution. If ADD sees two integers, it rewrites itself to ADD_INT — a single-byte in-place edit that eliminates all type checking on subsequent executions. If the types change later, it reverts to generic and re-specializes. This is what CPython now does, but UBI's fixed 32-bit instruction format makes the rewrite a single aligned byte write instead of a complex bytecode surgery.
