JavaScript — TimerPragma
function Timer() {
  this.count = 0;
  var self = this;
  setInterval(function() {
      self.count++;
      if (self.count > 10) {
          self.stop();
      }
  }, 1000);
  this.stop = function() {
      var self = this;
      clearInterval(function() {
          self.count = 0;
      });
  };
}
type Timer
  int count
end type

function tick(Timer <t>)
  t.count++
  if t.count > 10
    stop(t)
  end if
end function

function stop(Timer <t>)
  t.count = 0

Pragma is an independent research project. Here is what it is and why it exists.

The starting point was a practical frustration: systems code, web code, and general application code each live in separate languages with separate conventions, separate learning curves, and separate toolchains. Moving between them carries friction that has nothing to do with the underlying problem being solved. It is just the cost of the ecosystem as it exists.

At the same time, watching code generation models work — and more importantly watching them fail — made something clear. The failure mode is not that the models are insufficiently powerful. It is that natural language is a noisy, ambiguous input for producing deterministic output. The model interprets, guesses, fills gaps. The result varies. A mechanized approach to code generation — structured input that maps predictably to structured output — would produce something more reliable, more reproducible, and easier for a person to read and verify without needing to understand the full underlying implementation.

Those two observations together are what Pragma and its broader conceptual framework, USAMS — Unified Syntax and Architectural Model for Software, are an attempt to address. One syntax that covers the range from low-level systems work to high-level application development. Runtime behavior that is declared rather than inherited from whichever language you happened to pick. And a schema-based expression layer that describes what a program does in the same terms you would use to explain it to another person, rather than the terms required to build it in an IDE.

This is a hobby project and an ongoing design exercise, not a product. The work is documented as it develops. Feedback and collaboration from people who find this worth thinking about is genuinely welcome.

Different languages behave differently.

Rust already toggles safety with unsafe blocks. Inside unsafe, you get raw pointers, manual memory, no borrow checker. Outside, you get full safety guarantees. Same language, two modes, toggled per-block.

C# has unsafe context too — lets you use pointers and fixed-size buffers, opting out of GC for that scope.

D has @nogc annotation — marks functions that aren't allowed to allocate from the garbage collector. You can mix GC and manual memory in the same program.

Zig lets you choose allocators per-call. Want arena allocation here, general purpose there, no allocation at all in this hot path? You pass the allocator explicitly.

Swift has both ARC (reference counting) and unsafe pointers available in the same language.

But no language puts all of these on a single dial panel. Every one picks a default and lets you escape from it in limited ways. Pragma has no default — you declare your policy and the compiler enforces it.
memory: manual // or gc, or refcount, or arena safety: checked // or unchecked types: strict // or dynamic int_width: 32 // or 64, or platform concurrency: threads // or async, or cooperative

Every function that allocates, every data structure, every IO operation works under all configurations. A linked list under GC doesn't need a destructor. Under manual memory it does. Under refcount it needs weak reference support.

One standard library with three implementations of everything makes this a reality.

No novel ideas, no reinventing the wheel. The syntax decisions borrow from what already exists across C, Python, and others — the goal is to unify and minimize, not to invent. The runtime configuration layer formalizes choices that languages already make implicitly, just without giving you a say. And the schema expression layer applies the same pattern that worked in SQL, HTML, CSS, and JSON to general application programming.
