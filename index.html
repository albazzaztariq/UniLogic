Overview

Pragma is a statically-typed systems language that transpiles to C. It is designed to be readable to anyone who has written code in any modern language.

Every Pragma program has at least a main function. Functions, types, and constants are declared at file scope above it. This both acknowledges the older C language syntax and adds some clarity for beginners.

Program Structure

No wrapper blocks. File scope holds all declarations. main is the entry point.

function main() returns int
  // entry point
end function
Note main or MAIN are both accepted as the entry point name.

Comments

// full line comment
int x = 10  // inline comment

Imports

import Math
import Collections

String

A first-class type. The underlying char array size is hidden — visible on hover in the IDE (LSP). When compiling to C, the compiler performs flow analysis to determine the maximum size the string can reach across the program. For runtime inputs, allocation is bounded to a safe maximum.

string name     = "hello"
string greeting = "Hi " + name
Note Memory safety cannot be guaranteed for runtime string inputs (user input, file reads). The compiler will warn when this is the case.

Collections

Array

Fixed-size. Size declared after the type keyword. Zero-indexed.

array int[5] a = [1,2,3,4,5]
a[2]              // 3

array int a       // no declared size — decays to pointer

// Multi-dimensional: rows then columns
array int[5][3] grid
grid[0][2] = 42

List

Dynamic, resizable. Capacity hint in brackets.

list string[3] names
list string[3][2] grid   // 2D list

Map

Key-value store with optional capacity hint.

map [string,int][50] wc   // string→int, capacity 50

wc["key"]           // read  (calls wc_get)
wc["key"] = val     // write (calls wc_set)
wc_has("key")       // 1 if key exists, 0 otherwise
wc_remove("key")   // delete a key

Casting

Use change(expr)->Type. Float-to-int truncates — no rounding.

int n    = change(3.9)->int      // 3 — truncates
string s = change(42)->string    // "42"
float g  = change(10)->float     // 10.0
bool b   = change(1)->bool       // true

If / Else

if score > 200
  print "high score"
else if score > 100
  print "good score"
else
  print "keep trying"
end if

While

int count = 0
while count < 5 do
  count++
  print count
end do

For

Two forms — use whichever reads better.

// expressive
for int i = 0 while i < 10 do i++
  print i
end for

// C-style
for i=0, i<10, i++
  print i
end for

For Each

list string[3] names = ["Alice", "Bob", "Carol"]

for each name in names
  print name
end for

Match — no fall-through

Colon on same line for single statements. Colon then newline for multi-statement cases.

int code = 2

match code
  1: print "one"
  2: print "two"
  3: print "three"
  default: print "other"
end match

Iterate — fall-through

Execution continues into the next case unless escape stops it.

iterate code
  1: print "one"
  2: print "two"
     escape
  3: print "three"
     escape
end iterate
Coming from C iterate is C's switch. Fall-through by default — identical behaviour. escape is break.

Functions

function add(int a, int b) returns int
  return a + b
end function

function greet(string name) returns none
  print "Hello, " + name

function factorial(int n) returns int
  if n <= 1
    return 1
  end if
  return n * factorial(n - 1)
end function

Note: return types must be declared just like with variables. Python leaves this open, Pragma makes it clear.

Types

type Player
  string name
  int age
  float score = 0.0
end type

type ProPlayer inherits Player
  int rank
end type

Player p = new Player["Alex", 25, 98.5]
print p.name
Coming from C type is a struct. Plain field declarations compile to an identical memory layout. Methods and inheritance are additive — ignore them and a type is exactly a struct.

Fixed — Enumerations

Named integer constants. Access as Name.member.

fixed Priority
  low    = 1
  medium = 2
  high   = 3
end fixed

int p = Priority.high    // p = 3
Coming from C fixed is enum. Access is dot notation — Priority.high instead of the typical PRIORITY_HIGH convention.

Print

print "hello world"

Try / Catch

Throw with error. Multiple exception types in one catch use or.

error ExcType "message"    // throw with message
error ExcType              // throw without message

try
  // risky code
catch(IOException err)
  print "IO error"
end catch
catch(RuntimeError or ValueError err)
  print "runtime error"
end catch
end try

Base Pragma Reference

Low-level operations and systems work. These concepts may not exist or be as easily accessible in Python, Java, or other modern languages. The angle brackets < > are the signal that you are working at the memory level.

Why Base Pragma?

Pointers

A pointer holds a memory address. Declared as type <name>. The type tells the compiler how many bytes to read at that address and how to interpret them.

int x = 42
int <p> = address(x)      // p holds the address of x

int8 b = 7
int8 <bp> = address(b)    // 8-bit pointer
The angle bracket rule Any operation involving a raw memory address uses < >. This is the consistent visual signal across all of Base Pragma — not capitalisation, not special syntax, just the brackets.
Coming from C int <p> = address(x) is int *p = &x in C. The angle brackets wrap the pointer name, and address() replaces &.

address and deref

PragmaC equivalentMeaning
address(x)&xTake the address of plain variable x
address(<p>)&pTake the address of pointer p
deref(<p>)*pRead the value at the address in p
int val = 99
int <p> = address(val)
int copy = deref(<p>)             // copy = 99
deref(<p>) = 100                  // write through pointer — val is now 100
Coming from C deref(<p>) is *p in C. Reading: int copy = deref(<p>) is int copy = *p. Writing: deref(<p>) = 100 is *p = 100.

mem — Raw Address Access

Read or write at any numeric address. Used for memory-mapped hardware registers.

uint32 <reg> = address(someVar)
mem(<reg>)                          // read at address reg
mem(<0xDEAD0000>)                  // read at hard-coded address
mem(<0xDEAD0000>) = 1              // write to that address

Pointer Arithmetic

Use <p> + n to advance a pointer by n elements. Units are the pointer's element type — not raw bytes.

int <p> = address(arr)
<p>[0]        // element 0 — same as deref(<p>)
<p> + 5        // element 5 — like *(p+5) in C
<p> + (i + 2)  // element i+2
<p> + 3        // advance 3 from current position

size

Byte size of a type. Equivalent to sizeof in C.

int s1 = size(int64)    // 8
int s2 = size(float)     // 4
int s3 = size(uint8)    // 1

none — Generic Pointer

A pointer with no specific type. Equivalent to void* in C. Use when a function must accept or return a pointer of any type.

none <generic> = address(anyVar)

// Assign to a typed pointer — the declaration carries the type:
int <typed> = generic

Bitwise Operations

Operate at the individual bit level. Use uint types for bit manipulation — applying these to signed integers flips the sign bit and produces negative results.

PragmaCMeaning
both1&Bitwise AND — output bit is 1 only if both input bits are 1
both0~(a|b)Bitwise NOR — output bit is 1 only if both input bits are 0
either1|Bitwise OR — output bit is 1 if either input bit is 1
delta^Bitwise XOR — output bit is 1 if the bits differ
bitflip x~xBitwise NOT — inverts every bit
negate x-xArithmetic negation
x left nx << nLeft shift — multiply by 2ⁿ
x right nx >> nRight shift — divide by 2ⁿ (logical on uint)
left=<<=Left shift assign
right=>>=Right shift assign
uint32 flags = 0
flags = flags either1 4      // set bit 2
flags = flags both1 0xFF     // mask to low byte
flags = flags delta 1         // toggle bit 0
flags = flags left 2          // shift left 2 positions
flags = flags right 1         // shift right 1 position
flags = bitflip flags          // invert all bits
Use uint types for bit manipulation bitflip on a signed int flips the sign bit. bitflip 5 gives -6, not 250. Always use uint8, uint16, uint32, or uint64 when working with bits.

Bit Fields

Declare a field with a specific number of bits using type.bits=N fieldname syntax. Fields pack into adjacent bits in declaration order.

type PackedHeader
  uint.bits=1 is_valid    // 1 bit
  uint.bits=7 count       // 7 bits — total so far: 1 byte
  uint32 value            // 4 bytes, no bit constraint
end type
Note Bit field groups should total to a byte boundary (8, 16, 32, or 64 bits) for predictable cross-platform layout.

pack() — Type Packing

By default the compiler may insert padding between fields to align them in memory. pack() removes all padding — every field follows immediately after the previous one with zero gaps. Required for network packet headers and file format types where byte position must be exact.

pack(PackedHeader)
Warning Packed types can cause misaligned memory access on some processor architectures, triggering a hardware fault or a silent performance penalty. Only use when exact byte layout is required.
Coming from C pack(TypeName) is __attribute__((packed)) or #pragma pack(1) in C. Same effect — removes struct padding for exact byte layout.

Maps directly to C's char[]. Size, allocation, and freeing are your responsibility.

Why Base Pragma?

Not every program needs to manage memory, work with pointers, or think in bits. Most application code never touches any of this. Separating it into its own file type means beginners and those who don't need these details are never forced to encounter them. A .run file stays clean. A .base file signals immediately that something lower-level is happening. That separation makes programming more comfortable for people who aren't ready for it, and clearer for those who are.

Coming from C

If you already know C, Base Pragma maps directly onto what you know. The semantics are the same — only the syntax changes. The goal is maximum readability without sacrificing any low-level control.

type vs struct In C, a struct is a named group of typed fields — nothing more. A Pragma type is the same thing, with the option to add methods, inheritance, default field values, and constructors. If you ignore those features and just declare fields, it compiles to an identical struct. There is no separate struct keyword in Pragma because the only difference is capability, not concept.

Pragma was designed with one idea in mind: unity.
Single Expression, Easily Understood
One language that everyone can understand. Pragma uses natural language to make a program readable as sentences rather than mathematical expression and notation.
High or Low
Frictionless between modern high-level applications or low-level monolithic systems work.

The language transpiles to C and vice versa. Future work will interface with Python, Java, C#, and JavaScript. This will allow seamless comprehension across platforms.

Different languages behave differently.

Rust already toggles safety with unsafe blocks. Inside unsafe, you get raw pointers, manual memory, no borrow checker. Outside, you get full safety guarantees. Same language, two modes, toggled per-block.

C# has unsafe context too — lets you use pointers and fixed-size buffers, opting out of GC for that scope.

D has @nogc annotation — marks functions that aren't allowed to allocate from the garbage collector. You can mix GC and manual memory in the same program.

Zig lets you choose allocators per-call. Want arena allocation here, general purpose there, no allocation at all in this hot path? You pass the allocator explicitly.

Swift has both ARC (reference counting) and unsafe pointers available in the same language.

But no language puts all of these on a single dial panel. Every one picks a default and lets you escape from it in limited ways. Pragma has no default — you declare your policy and the compiler enforces it.
memory: manual // or gc, or refcount, or arena safety: checked // or unchecked types: strict // or dynamic int_width: 32 // or 64, or platform concurrency: threads // or async, or cooperative

Every function that allocates, every data structure, every IO operation works under all configurations. A linked list under GC doesn't need a destructor. Under manual memory it does. Under refcount it needs weak reference support.

One standard library with three implementations of everything makes this a reality.

There's no novel ideas, no reinventing the wheel: it allows anyone to write in C without the more unintuitive syntax of C, as well as understand other C programs directly in their language. If adopted more widely, it means everyone can understand directly each other's programs because syntactical style and rules are either expressed or familiar. And with the new heavier ML and AI workloads, it means hot paths and runtime become first-class citizens.
