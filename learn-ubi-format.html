Inside a .ubi File

The format, sections, and metadata the compiler provides to help the interpreter.

What's in the File

A .ubi file is not source code and not machine code. It is a compact binary format — a sequence of instructions that the UBI interpreter knows how to execute. You cannot read it in a text editor any more than you can read a JPEG by opening it in Notepad. It is meant for the interpreter, not for humans.

The file is divided into sections, each with a specific purpose. Here is the high-level layout:

+--------------------------------------------------+
|  Header (24 bytes)                                |
|  Magic number, version, section sizes             |
+--------------------------------------------------+
|  Constant Pool                                    |
|  Numbers, strings, booleans used by the program   |
+--------------------------------------------------+
|  Function Table                                   |
|  Where each function starts, register count       |
+--------------------------------------------------+
|  Block Metadata                                   |
|  Which parts of the code are hot (run frequently) |
+--------------------------------------------------+
|  Control Flow Table                               |
|  Which blocks can jump to which other blocks      |
+--------------------------------------------------+
|  Instructions                                     |
|  The actual bytecode (fixed 32-bit instructions)  |
+--------------------------------------------------+
|  Debug Section (optional)                         |
|  Variable names, line numbers, types              |
+--------------------------------------------------+

If you have worked with Java, this is the same idea as a .class file. If you have worked with Python, think .pyc. The concept is identical — compiled-but-not-native instructions bundled with metadata — but the specific format and what metadata is included differs significantly. UBI ships far more compiler intelligence in the file than either of those formats.

The Constant Pool

Programs reuse values. The number 0, the string "error", the boolean true — these appear over and over. Instead of encoding each value directly into every instruction that uses it, the compiler stores each unique value once in a table at the top of the file: the constant pool.

Instructions reference constants by index. When the interpreter sees "load constant #3 into register r0," it goes to the 4th entry in the constant pool (indices start at 0) and copies that value into the register. The instruction itself is just a 32-bit word containing the opcode and the index — no matter whether the constant is the integer 42 or a 500-character string.

This is what keeps the instruction stream compact. Every instruction is exactly 32 bits. The constants can be any size, but the instructions that reference them are always the same width. Uniform instruction width means the interpreter can jump to any instruction by offset without scanning — instruction N is always at byte offset N × 4.

Block Metadata and Hotness Counters

The compiler divides each function into basic blocks — straight-line chunks of code with no branches in the middle. A basic block runs from start to finish every time it executes; you cannot jump into the middle or branch out partway through. This is the fundamental unit of control flow analysis.

Each block carries a 6-bit counter (values 0 through 63) that tracks how many times it has been executed. This is the hotness counter. When a counter saturates — hits 63 — the interpreter knows this code is hot. It runs frequently and deserves extra optimization effort. In the future, saturating the counter will be the signal for a JIT compiler to compile that block to native machine code.

The key insight is that you do not want to optimize all code. In most programs, the vast majority of code runs once or twice — startup logic, error handlers, configuration parsing. Only the inner loops and hot paths deserve the investment of analysis and optimization. The hotness counter is what separates the two, and the compiler provides the infrastructure for it so the interpreter does not have to build it at runtime.

Did you know? The "magic number" at the start of a .ubi file is the bytes U, B, I followed by a null byte (0x55 0x42 0x49 0x00). Every binary format has one — it is how programs quickly check "is this file the right type?" without reading the whole thing. Java's is 0xCAFEBABE. PNG images start with 0x89504E47. Even GIF files start with the literal text GIF89a.

The Control Flow Table

Normally, an interpreter figures out "where can this code jump to?" by scanning the instructions at runtime. It sees a branch instruction, looks at the target offset, and follows it. UBI's compiler pre-computes this entire graph and ships it in the file as the control flow table.

Each entry in the table is an edge: "block 3 can jump to block 7 (branch taken)" or "block 3 falls through to block 4 (branch not taken)." Together, these edges form the complete control flow graph of the program. The interpreter gets this information for free the moment it loads the file.

Why does this matter? Two reasons. First, the interpreter can prefetch the next block's instructions into the CPU cache before it finishes executing the current block, because it already knows which blocks are reachable. Second, a future JIT compiler gets the entire control flow graph without performing any analysis at all — no dominator trees to compute, no loop detection to run. The compiler already did that work. The JIT just reads the table and starts optimizing.

The Debug Section (and Stripping)

By default, the compiler includes a debug section at the end of the file: variable names, function signatures, source line numbers, and type information. This is what makes error messages useful. With debug info, you get: "Error at line 42 in function 'calculate': division by zero." Without it: "Error at instruction offset 0x01A4."

For distribution, you can strip the debug section entirely:

ul build app.ul -t ubi --strip

The program still runs identically — the interpreter does not need any of this information to execute the bytecode. But decompiling a stripped file gives you generic names like _r0, _r1 instead of total, count. The logic is all there; the meaning is gone.

It is like shipping a building without the blueprints. It still stands, but good luck figuring out which wall is load-bearing.

Why This Design

The JVM (1995) kept its bytecode simple and put all intelligence in the runtime. HotSpot, the JVM's JIT compiler, discovers control flow, profiles hotness, and builds optimization data entirely at runtime. This worked — brilliantly — but it means every program pays a warmup cost while the runtime rediscovers facts the compiler already knew.

UBI does the opposite. The compiler does extra work at compile time to embed metadata — block hotness counters, control flow edges, register liveness — directly in the .ubi file. The interpreter can exploit this information from the very first instruction, with zero warmup.

This is the same approach Android's ART runtime and WebAssembly use. It is the modern consensus: if the compiler already knows something, do not make the runtime rediscover it. Ship the knowledge in the file. Let the interpreter be fast from the start, and give the JIT a head start when it kicks in later.

Did you know? A typical UL program produces a .ubi file that is 2–5x smaller than the equivalent .pyc (Python bytecode) file, because UBI's fixed 32-bit instructions are more compact than Python's variable-width bytecode. Smaller file means faster loading means faster startup.
