AI & ML Workload Optimization

This learning module covers the concepts underlying current optimization efforts for AI and ML workloads. Tools exist in C, Python, and other languages, but require specialist knowledge to properly use or involve multiple pieces of software rather than simple installation allowing use.

Some of the ideas necessary to operate — or better operate — tools in this domain are explained in the following sections. These are important to UL specifically, even more than other languages, because UL incorporates many of these optimizations as language-level features rather than library-level workarounds.

No prior knowledge of CPU architecture, GPU computing, or compiler internals is assumed. Each concept builds on the previous one.

Section 2 of 20

What UL Does Differently

UL is designed to incorporate modern AI and ML optimization as part of the language itself. This does not mean it bundles libraries you can already download with other languages, nor just standard library functionality. It refers to optimizations which must be explicitly applied — painstakingly, repeatedly, with much effort — because of the structure of other languages and their library implementations.

For example, mixed floating-point precision is offered through libraries in C and Python (PyTorch's .half(), NVIDIA's AMP), but requires function calls inserted at every possible position. This needs code analysis to identify insertion points and then placing those calls repeatedly. UL lets you declare it once as a runtime or compile-time setting and it's applied implicitly.

Likewise: sparsity as a construction and quantization as a type are language features that no other language offers natively. Prefetch annotation that's painless is built into the language itself. These exist at both the compiler and language level.

Section 4 of 20

From Code to Operations: LLVM

LLVM is the first layer at which your code is no longer code anymore — it's operations. The logic you wrote, the if/else branches, the function calls — these are reduced to a flat sequence of low-level instructions: load a value from memory, add two numbers, compare, branch. The structure and intent of your program is largely gone.

LLVM takes the compiler's output (called Intermediate Representation, or IR) and transforms it through dozens of optimization passes: removing dead code, inlining small functions, unrolling loops, reordering instructions for better CPU utilization. It then emits machine code for the specific CPU you're targeting.

This is where most performance is won or lost. The code you write is a suggestion. What LLVM produces after optimization is what actually runs. Understanding this layer helps you write code that optimizes well rather than code that fights the optimizer.

Section 5 of 20

Inside the CPU

A CPU processes instructions using several major components working together:

Registers — tiny, extremely fast storage slots directly inside the CPU. A modern x86-64 CPU has 16 general-purpose registers (64 bits each), 16 SSE/AVX vector registers (128–512 bits each), and several special-purpose registers (instruction pointer, flags, stack pointer). Data must be in a register before the CPU can operate on it.

ALU (Arithmetic Logic Unit) — performs integer arithmetic and logic operations. Modern CPUs have multiple ALUs that can execute several operations per clock cycle.

FPU (Floating Point Unit) — handles floating-point math. Separate from the ALU, with its own register file on older architectures.

Cache hierarchy — L1 (fastest, smallest, ~32KB per core), L2 (larger, ~256KB–1MB), L3 (shared across cores, ~8–32MB). Data flows from RAM → L3 → L2 → L1 → registers. Each step closer is ~10x faster.

Section 6 of 20

How Instructions Flow

The CPU processes each instruction through a pipeline of stages. Modern CPUs overlap these stages so multiple instructions are in-flight simultaneously:

Fetch — read the next instruction from memory (ideally from L1 instruction cache).

Decode — figure out what the instruction does, which registers it needs, what operation to perform.

Execute — the ALU/FPU performs the operation.

Memory access — if the instruction loads from or stores to memory, this happens here.

Writeback — the result is written to the destination register.

A modern CPU pipeline is 15–20 stages deep. When everything flows smoothly, one instruction completes per clock cycle (or more, with superscalar execution). When something disrupts the pipeline — a branch misprediction, a cache miss — the CPU stalls and the pipeline drains. This is where performance cliffs come from.

Section 7 of 20

Branch Prediction

Every if statement in your code becomes a branch instruction. The CPU must decide which path to take before it knows the answer, because the pipeline is 15+ stages ahead of execution. It guesses using a branch predictor — hardware that tracks the history of each branch and predicts the likely direction.

When the prediction is correct (95%+ of the time for well-structured code), execution continues at full speed. When it's wrong, the CPU must flush the pipeline — discard all the speculative work it did on the wrong path and restart from the correct branch. This costs 10–20 cycles per misprediction.

This is why sorted data processes faster than unsorted data for the same algorithm. The branch predictor can learn the pattern of sorted comparisons. Random data means random branches means constant mispredictions. In AI workloads with tight inner loops, a single poorly-predicted branch in the hot path can cost more than the actual computation.

Section 8 of 20

SIMD & Vector Registers

SIMD (Single Instruction, Multiple Data) lets one instruction operate on multiple values simultaneously. Instead of adding two numbers, you add 4, 8, or 16 numbers in a single operation using wide vector registers.

SSE — 128-bit registers. Process 4 floats or 2 doubles at once.

AVX/AVX2 — 256-bit registers. Process 8 floats or 4 doubles at once.

AVX-512 — 512-bit registers. Process 16 floats or 8 doubles at once.

This is the hardware foundation of vectorized computation. Neural network inference is almost entirely matrix multiplication — thousands of multiply-accumulate operations on arrays of floats. SIMD turns what would be N sequential operations into N/8 or N/16 vector operations. The compiler can auto-vectorize some loops, but getting the memory layout right (contiguous, aligned data) is the developer's responsibility.

Section 9 of 20

Cache: Hits, Misses, and Memory Layout

When the CPU needs data, it checks L1 first (~1 cycle), then L2 (~4–10 cycles), then L3 (~20–40 cycles), then RAM (~100–300 cycles). A cache miss at L1 that hits RAM costs 100–300x more than a cache hit. In tight loops, cache misses dominate execution time.

Data is loaded in cache lines — 64 bytes at a time. If you access one element of an array, the next 15 elements come free (assuming 4-byte floats). This is why iterating an array sequentially is fast and iterating a linked list is slow — arrays have spatial locality, linked lists don't.

Memory layout matters. A struct-of-arrays (SoA) layout where all X values are contiguous, then all Y values, then all Z values, vectorizes perfectly. An array-of-structs (AoS) where each point has X,Y,Z interleaved wastes cache lines loading Y and Z when you only need X. For AI workloads processing millions of tensors, this difference is the difference between saturating memory bandwidth and wasting 60% of it.

Section 10 of 20

The GPU Advantage

A CPU has 8–16 powerful cores optimized for sequential, complex work. A GPU has thousands of small cores optimized for doing the same simple operation on many data points simultaneously. Neural network training and inference are almost entirely parallel math — multiply matrices, add biases, apply activation functions — the same operation repeated millions of times across different data.

Tensor cores (NVIDIA) are specialized hardware units within GPU cores that perform matrix multiply-accumulate operations in a single clock cycle. A single tensor core can compute a 4x4 matrix multiply in one operation — something that takes dozens of instructions on a regular GPU core. This is purpose-built for deep learning.

The cost: data must be transferred from CPU memory to GPU memory over the PCIe bus. This transfer is slow relative to computation. Optimization often focuses on minimizing data movement — keeping tensors on the GPU, fusing multiple operations into one kernel to avoid round-trips, and overlapping computation with data transfer.

Section 11 of 20

Quantization

Neural network weights and activations are typically stored as 32-bit floating-point numbers (FP32). Quantization reduces this precision — to FP16 (half precision), BF16 (brain float), INT8, or even INT4 — trading accuracy for speed and memory savings.

Why it works: neural networks are inherently tolerant of approximation. The difference between a weight of 0.73291016 and 0.73 is irrelevant to the model's output in most cases. Reducing precision by half means twice as many values fit in cache, twice as many can be processed per SIMD instruction, and memory bandwidth requirements are halved.

In practice: INT8 quantization can provide 2–4x speedup with less than 1% accuracy loss for most inference workloads. The challenge is calibrating the quantization parameters — figuring out the right scale and zero-point for each layer so the reduced precision doesn't distort the output. In UL, quantization is a type-level declaration, not a per-tensor function call.

Section 12 of 20

Mixed Precision & FMA

Mixed precision runs computation in FP16 (fast, fits more in cache and SIMD registers) but accumulates results in FP32 (prevents precision loss from compounding over thousands of operations). This is the standard for modern training — NVIDIA's Automatic Mixed Precision (AMP) in PyTorch does this, but requires wrapping your training loop in a context manager and calling .half() on tensors selectively.

FMA (Fused Multiply-Add) computes a * b + c in a single instruction with a single rounding step instead of two separate operations with two rounding steps. This is faster and more accurate. Modern CPUs and GPUs have dedicated FMA units. The compiler can emit FMA instructions when it recognizes the pattern, but only if the code structure allows it — if the multiply and add are in separate statements with an intermediate variable, the compiler may not fuse them.

UL declares mixed precision as a compile-time setting. FMA patterns are recognized and fused automatically by the compiler without requiring specific code structure.

Section 13 of 20

Vectorization & Prefetch

Auto-vectorization is when the compiler transforms a scalar loop (processing one element at a time) into a vector loop (processing 4/8/16 elements at a time using SIMD). This requires: contiguous memory access, no data dependencies between iterations, no branches in the loop body, and aligned data. If any condition isn't met, the compiler falls back to scalar code silently.

Prefetch tells the CPU to start loading data from memory before the code actually needs it, hiding the memory latency. In C, this is __builtin_prefetch(addr) — a compiler intrinsic that must be placed manually at the right distance ahead of the access (too early wastes cache, too late doesn't help). Getting the prefetch distance right requires profiling on the specific hardware.

UL provides prefetch as a language-level annotation. The compiler calculates the optimal prefetch distance based on the target architecture's cache latency, and inserts prefetch instructions automatically for annotated data structures. No manual tuning, no hardware-specific intrinsics.

Section 14 of 20

Sparsity & Kernel Fusion

Sparsity exploits the fact that many neural network tensors are mostly zeros (sometimes 80–95% zeros after pruning). Instead of multiplying by zero thousands of times, skip those operations entirely. NVIDIA's Ampere and Hopper GPUs have hardware support for 2:4 structured sparsity — for every 4 elements, at least 2 must be zero, and the hardware skips them automatically.

In Python/C, exploiting sparsity requires converting dense tensors to sparse formats (CSR, COO), using sparse-specific library functions, and managing the conversion overhead. In UL, sparsity is a type-level construct — declare a tensor as sparse and the compiler handles format selection and operation dispatch.

Kernel fusion combines multiple GPU operations into a single kernel launch. Without fusion, each operation (matmul, bias add, ReLU activation) launches a separate GPU kernel with its own memory read/write cycle. Fused kernels read input once, perform all operations, and write output once — eliminating intermediate memory traffic. Libraries like PyTorch's torch.compile do this automatically now, but the fused kernel quality depends heavily on the compiler's ability to analyze the operation graph.

Section 15 of 20

Memory: NUMA, Arenas, and Huge Pages

NUMA (Non-Uniform Memory Access) is a server memory architecture where each CPU socket has its own local memory. Accessing local memory is fast; accessing another socket's memory crosses an interconnect and is 2–3x slower. Enterprise servers with 2+ sockets are NUMA by default. If your application's threads are on socket 0 but the data is in socket 1's memory, every access pays the cross-socket penalty. NUMA-aware allocation pins data to the socket that will use it.

Huge pages reduce TLB (Translation Lookaside Buffer) misses. Normal memory pages are 4KB; the CPU's TLB can only cache a limited number of page mappings. For workloads with large memory footprints (multi-GB models), 4KB pages mean constant TLB misses. Huge pages (2MB or 1GB) cover more memory per TLB entry, drastically reducing misses. Linux supports this via madvise or transparent huge pages; it's rarely enabled by default.

Arena allocation and specialized allocators like mimalloc (Microsoft's high-performance malloc replacement) reduce allocation overhead for workloads that allocate and free many small objects rapidly — typical in inference pipelines processing batches of requests.

Section 16 of 20

Compiler Optimizations

PGO (Profile-Guided Optimization) — compile the program, run it on representative input, feed the profile data back to the compiler. The compiler uses actual execution frequencies to decide what to inline, how to lay out code in memory, and which branches are hot. Typically 10–20% speedup for real workloads. Two-phase build process.

Peephole optimization — the compiler scans small windows (2–5 instructions) and replaces patterns with faster equivalents. Example: x * 2 becomes x << 1, or x * 0 is eliminated entirely.

Loop nest optimization — transforms nested loops for better cache behavior. Loop tiling (blocking) processes a tile of data that fits in L1 cache before moving to the next tile, instead of streaming through the entire dataset. Loop interchange swaps the order of nested loops so the inner loop accesses contiguous memory.

Operator fusion — at the compiler level (not GPU kernel level), combines adjacent operations that share data. A multiply followed by an add on the same result becomes a single FMA instruction. A load followed by an immediate use can be combined into a load-and-operate instruction.

Section 17 of 20

IR Patterns & Scheduling

Multi-level IR — modern compilers use multiple intermediate representations at different abstraction levels. MLIR (Multi-Level IR, from Google/LLVM) represents operations at the tensor level, the loop level, and the machine level in a single framework. This lets optimizations happen at the right level of abstraction — kernel fusion at the tensor level, vectorization at the loop level, register allocation at the machine level.

Equality saturation — instead of applying optimization rules one at a time (which can miss the global optimum), equality saturation explores all possible rewrites simultaneously and selects the best. Tools like egg (e-graphs good) use this for compiler optimization. It finds optimizations that sequential rewriting misses.

Halide's schedule concept — Halide separates what to compute from how to compute it. You write the algorithm once, then write a separate schedule that controls tiling, vectorization, parallelism, and memory layout. This lets you explore hundreds of optimization strategies without changing the algorithm. TVM and other ML compilers adopt this idea for neural network operators.

Static operation scheduling — the compiler reorders instructions to maximize instruction-level parallelism, minimize pipeline stalls, and hide memory latency. On in-order CPUs (common in embedded/mobile), this is critical — the compiler must order instructions so that data is ready when the next instruction needs it.

Section 18 of 20

Library Landscape

PyTorch internals — at its core, PyTorch dispatches tensor operations to optimized C++/CUDA kernels. torch.compile (TorchDynamo + TorchInductor) captures the Python-level operation graph, optimizes it, and generates fused CUDA kernels. This is Python compensating for not having these optimizations in the language itself.

CUDA kernels — custom GPU programs written in CUDA C/C++ for operations where library kernels aren't fast enough. Writing a custom CUDA kernel requires understanding GPU thread hierarchy (grids, blocks, warps), shared memory management, and memory coalescing. Libraries like Triton (from OpenAI) simplify this by letting you write GPU kernels in Python-like syntax.

mimalloc — Microsoft's high-performance allocator. Drop-in replacement for malloc. Uses per-thread free lists, small page sizes, and eager decommitting. Typically 5–15% faster than glibc malloc for allocation-heavy workloads. No code changes needed — just LD_PRELOAD on Linux.

Go's split stacks / segmented stacks — goroutines start with tiny stacks (2–8KB) that grow on demand. This allows millions of concurrent goroutines without pre-allocating large stacks. Relevant to UL's concurrency model — lightweight execution contexts for async workloads.

Section 19 of 20

UL's Built-in Tools

UL incorporates these optimizations as language and compiler features rather than library-level workarounds:

Mixed precision — declare via DR setting. The compiler handles FP16 computation with FP32 accumulation automatically. No .half() calls.

Quantization — a type-level declaration. int8 and other reduced-precision types are first-class. The compiler selects quantization parameters.

Sparsity — a type-level construct. Declare a tensor as sparse, the compiler selects the format and dispatches sparse-aware operations.

Prefetch — language-level annotation. The compiler calculates optimal prefetch distance per target architecture.

Memory layout — controllable via DR settings. SoA vs AoS, alignment, padding — declared, not manually managed.

FMA — automatic pattern recognition and fusion by the compiler.

Vectorization — compiler auto-vectorizes with target-aware SIMD width selection. DR settings control whether to prefer AVX2 or AVX-512.

Kernel fusion, PGO, loop nest optimization — available through compiler flags and normalization levels.
