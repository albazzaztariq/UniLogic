Dynamic Runtime — DR

Dynamic Runtime

UniLogic exposes 9 runtime settings that cover the decisions most languages make once at the language level and never revisit. Each setting is declared at the top of a source file and enforced by the compiler for that file. All settings have defaults — if you don't declare them, the defaults apply silently.

Runtime: Basics and Why It Matters
Not required, defaults apply. Programs run without ever touching DR.
If you aren't familiar with Runtime Environments, we recommend going to the link above on Runtime Basics and reviewing those materials first.

Most languages pick a memory model and an error strategy at design time. You inherit them when you choose the language, and working around them — if possible at all — requires unsafe blocks, external libraries, or runtime flags. UL treats these as per-file compile-time configuration. The standard library is implemented to work correctly under all combinations.

memory: gc // or: manual · refcount
safety: checked // or: unchecked
int_width: 32 // or: 64 · platform
float_model: ieee // or: fast
overflow: trap // or: wrap · saturate
types: strict // or: dynamic
allocator: system // or: custom · embedded
bounds_check:enabled // or: disabled
stack_size: default // or: N (in KB, e.g. 512)

memory

Controls how the program allocates and frees heap memory. This setting affects every allocation in the file. The standard library's collection types, strings, and data structures all work under any model.

memory gc  |  manual  |  refcount

gc — Garbage collected. The runtime tracks live objects and reclaims unreachable memory automatically. Lowest programmer effort; some pause latency.

manual — Programmer manages allocation and deallocation. Use malloc, free. Full control, zero overhead, full responsibility.

refcount — Reference counting. Objects are freed when their reference count drops to zero. Deterministic timing; cycles require explicit handling.

default: gc
Sub-option — memory: gc

Controls which garbage collection strategy the runtime uses. Performance-sensitive code may care significantly about pause behaviour and collection timing.

gc_strategy generational  |  tracing  |  incremental

generational — Divides objects into generations by age. Short-lived objects are collected frequently in a cheap minor pass; long-lived objects are promoted and collected rarely. Best general-purpose choice — low average pause time, high throughput.

tracing — Traces all live objects from roots on each collection cycle. Simpler and fully accurate, but must scan the entire heap. Predictable behaviour; pauses scale with heap size.

incremental — Breaks collection work into small slices interleaved with program execution. No single large pause — collection cost is spread across time. Best for latency-sensitive applications such as games or real-time systems.

default: generational  ·  only applies when memory is gc

allocator

Selects the underlying allocator used for heap operations. Applies when memory is manual or gc.

allocator system  |  custom  |  embedded

system — Uses the platform allocator (malloc/free on POSIX, HeapAlloc on Windows).

custom — Binds to a user-provided allocator struct implementing the allocator interface. Useful for memory pools, instrumentation, or replacing jemalloc/mimalloc.

embedded — Statically-sized allocator with no dependency on the OS heap. For bare-metal and firmware targets with no dynamic memory subsystem.

default: system
Contract note — allocator: custom

When allocator: custom is set, the provided struct must implement the UL allocator interface: alloc(size, align) → ptr, free(ptr), and realloc(ptr, new_size) → ptr. This is not a setting — it is a compile-time contract. The compiler will reject a custom allocator that does not satisfy the interface.

overflow

Determines what happens when an integer arithmetic operation exceeds the type's range.

overflow trap  |  wrap  |  saturate

trap — Overflow triggers a runtime abort. Equivalent to Rust's debug builds. Safe but halts execution.

wrap — Overflow wraps around (two's complement). Identical to C's unsigned arithmetic and Rust's release builds. Fastest; behaviour is defined but may be unintended.

saturate — Overflow clamps to the type's maximum or minimum value. Common in signal processing and audio code.

default: trap

stack_size

Sets the initial thread stack size. Relevant for recursive algorithms, large stack frames, or embedded targets with constrained memory.

stack_size default  |  N (KB)

default — Platform default (typically 1–8 MB depending on OS).

N — Explicit size in kilobytes. Example: stack_size: 256 allocates a 256 KB stack.

default: default

safety

Controls whether the compiler inserts runtime safety checks. Does not affect formal verification — that is a separate compilation step controlled by the MemProof pipeline.

safety checked  |  unchecked

checked — Null pointer dereference, out-of-bounds access, and type mismatch checks are inserted at compile time. Violations abort with a diagnostic message.

unchecked — No runtime checks. Behaviour on violation is undefined. Use in production hot paths where you have verified safety separately.

default: checked

bounds_check

Specifically controls array and pointer bounds checking, independently of safety. Useful when you want all other safety checks but need tight loops to run without bounds overhead.

bounds_check enabled  |  disabled

enabled — Every array access and pointer offset is range-checked at runtime.

disabled — Bounds checks are omitted. The compiler will warn if this is set without safety: unchecked.

default: enabled

int_width

Sets the width of the unqualified int type. Does not affect explicitly-sized types (int32, int64, etc.).

int_width 32  |  64  |  platform

32 — int is always 32-bit signed, regardless of target architecture.

64 — int is always 64-bit signed.

platform — int matches the platform's native integer width. Equivalent to C's int; not recommended for portable code.

default: 32

float_model

Controls floating-point semantics.

float_model ieee  |  fast

ieee — Strict IEEE 754 compliance. Results are fully reproducible across platforms. Required for numerical software, financial code, and anything where floating-point output must be deterministic.

fast — Allows the compiler to reorder, fuse, and reassociate floating-point operations for performance. Equivalent to -ffast-math. Results may differ by small amounts across architectures.

default: ieee

types

Controls whether the compiler enforces strict static typing or allows dynamic type behaviour. This setting is not target-independent — the valid values depend on the compile target. Declaring types: dynamic for a target that cannot support it is a compile error, not silent incorrect output.

types strict  |  dynamic

strict — All type coercions are explicit. Mixing int and float in an expression is a compile error without a cast. Valid for all targets.

dynamic — Common numeric promotions are inserted automatically (int → float, narrower → wider). Closer to how Python or JavaScript handle arithmetic. Not valid when targeting C — compile error.

default: strict
Target constraints

The C target has a static type system that does not support dynamic typing. The compiler cannot emit valid C from dynamically typed source. Declaring types: dynamic when targeting C is a compile error. The Python and JavaScript targets support both modes.

concurrency

Controls which concurrency model the compiler targets when emitting code. Different targets expose concurrency through different mechanisms — declaring this setting tells the compiler which model to emit.

This setting does not gate whether @async and @await are available in source — those decorators work regardless. It controls what the emitted runtime code looks like.

concurrency parallel  |  threaded  |  async

parallel — CPU-bound data parallelism with shared memory. Maps to OpenMP or platform thread pools. Not available on the JS/HTML target.

threaded — I/O-bound thread concurrency. For Python targets: threading module. For C targets: pthreads. Not available on the JS/HTML target.

async — Cooperative single-threaded event loop. For Python targets: asyncio. For JavaScript targets: native async/await. Not available on the C target.

default: threaded
Target constraints

parallel and threaded are not available on the JS/HTML target — JavaScript is single-threaded; declaring either for that target is a compile error. async is not available on the C target — C has no built-in event loop; declaring it for that target is a compile error.

Settings are declared in a comment block at the top of each source file. If a setting is omitted, the default listed above is used — no configuration required for standard code. Settings do not propagate across files; each file is compiled according to its own declarations.
