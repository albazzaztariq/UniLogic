Error Handling

Four models. You choose per function, per file, or per project.

Overview

UL does not force one error handling style. Different parts of a program have different needs — a math library should propagate errors cleanly, a CLI script just needs to exit with a code, and a quick prototype doesn't need error handling at all.

The four models are:

ModelWhen to use
Result typesLibraries, APIs, any function where the caller should decide what to do with an error
Exit codesScripts and CLI tools that report success/failure to a shell
Explicit exit()Fatal errors deep in the call stack where recovery isn't possible
none (void)Quick programs, prototypes, functions that can't fail

They are not mutually exclusive. A program can use Result types in its library functions, exit codes in main, and void returns in helper utilities — all in the same project.

Result Types

A function that can fail declares its return type as type|error. On success it returns ok value. On failure it returns error "message". The caller must handle both cases.

function divide(int a, int b) returns int|error
  if b == 0
    return error "division by zero"
  end if
  return ok a / b
end function

The ? operator

The ? operator propagates errors up the call chain automatically. If the expression evaluates to an error, the enclosing function immediately returns that error. If it evaluates to ok, the value is unwrapped.

function average(int a, int b, int count) returns int|error
  int sum = a + b
  int result = divide(sum, count)?   // if error, return it immediately
  return ok result
end function

function main() returns int
  int val = average(10, 20, 0)?     // propagates "division by zero"
  print val
  return 0
end function
Boundary behavior When ? is used in a function that does not return type|error (like main() returns int), an unhandled error prints to stderr and exits with code 1. The error does not silently disappear.

Exit Codes

When main returns int, the return value becomes the process exit code. The OS reads this value. 0 means success. Any non-zero value means failure. This is the same pattern as C's int main().

function main() returns int
  // do work
  if something_failed
    return 1   // general error
  end if
  return 0     // success
end function

Exit codes are only meaningful at the top level. A library function returning int is just returning an integer — it's not an exit code unless that value reaches the OS. Use this model when writing scripts or CLI tools that need to signal success/failure to a shell, a CI system, or a parent process.

Explicit exit()

exit(code) terminates the program immediately from anywhere in the call stack. It does not require main to declare a return type. It does not unwind the stack. It stops.

function load_config(string path) returns none
  if not file_exists(path)
    print "config not found: " + path
    exit(1)
  end if
  // load config...
end function

function main() returns none
  load_config("app.conf")
  // if we get here, config loaded successfully
end function

Maps to C's exit(), Python's sys.exit(), Java's System.exit(). Use for fatal errors where recovery is not possible or not worth the complexity — missing config files, invalid command-line arguments, failed initialization.

exit(int code) is a built-in function. It accepts an integer exit code and terminates the process.

void / none (No Error Handling)

If a function returns none and has no error path, that's valid. Not every function can fail. Not every program needs error handling.

function main() returns none
  print add(2, 3)
end function

The compiler inserts return 0 at the OS level automatically. No return statement is needed. This is the same as Python's implicit None return and Java's void main. Use for scripts, prototypes, and programs where errors are handled by crashing (which is a valid strategy for many programs).

What Other Languages Do

Languagemain signatureError handling
C int main() Exit code via return. Implicit return 0 since C99. exit() available anywhere. No built-in Result type.
Python No main function Exceptions for everything. sys.exit() for codes. No return-based error model.
Java void main() Always void. System.exit() for codes. Checked exceptions force handling. No Result type in stdlib.
JavaScript No main concept Exceptions. process.exit() in Node.js. Promises use reject/resolve. No OS-level exit code from browser JS.
Rust fn main() or fn main() -> Result Result type with ? operator. std::process::exit(). Panics for unrecoverable errors.
UL Your choice All of the above. Result types with ?, exit codes via int return, exit() from anywhere, or void with no error path. You choose per function.
