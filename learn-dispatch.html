Dispatch, Threading, and Superinstructions

The mechanics of how an interpreter turns bytecode into action — and why the naive way is slow.

The Dispatch Loop

An interpreter does the same thing over and over: read the next instruction, figure out what it means, do it, advance to the next one. That's the dispatch loop. It's the heartbeat of every bytecode interpreter ever built.

The "figuring out what it means" step is called dispatch, and it's where interpreters spend most of their time — not doing the actual work (adding numbers, loading variables), but deciding which work to do. The simplest way to implement dispatch is a switch statement:

while (running) {
    opcode = bytecode[pc++];
    switch (opcode) {
        case ADD:  reg[a] = reg[b] + reg[c];  break;
        case SUB:  reg[a] = reg[b] - reg[c];  break;
        case LOAD: reg[a] = constants[b];      break;
        case JMP:  pc = target;                break;
        // ... 100+ more cases
    }
}

This works, but it's slow. The reason is the CPU's branch predictor. Modern CPUs guess which instruction comes next so they can start executing it before the current one finishes. For a normal if/else, the predictor is excellent. But a single switch statement shared across 100+ opcodes? The predictor sees one branch point that could go 100 different directions. It's terrible at this. Every misprediction costs 15–20 CPU cycles — wasted work that has to be thrown away.

Did you know? The CPU's branch predictor is right about 95% of the time for normal code. But a switch statement with 100+ cases (like an interpreter dispatch) drops it to around 50% — basically a coin flip. That's why dispatch technique matters so much.

Threading Techniques

First, a clarification: "threading" here has nothing to do with multi-threading or parallelism. It's an older usage of the word — it means how instruction handlers are "threaded" together, connected in a chain, like beads on a string. The term predates multi-threaded programming by decades.

Direct threading (computed goto)

Instead of every handler jumping back to one central switch statement, each handler jumps directly to the next handler. Think of it like a relay race: each runner hands the baton straight to the next, instead of everyone running back to the starting line between legs. In C, this uses GCC's goto *label_table[opcode] extension. Because each handler is its own branch point, the CPU's branch predictor now has 100+ separate sites to learn patterns for, instead of one overloaded site. Performance improves 15–25% over a switch.

Context threading

Context threading takes a different approach entirely. Instead of using branch prediction (which is mediocre for interpreters), it uses the CPU's return address predictor — a dedicated piece of hardware designed to predict where call/return pairs will go. This predictor is extremely accurate because call/return patterns are inherently more predictable than arbitrary jumps. Each bytecode handler becomes a tiny function that gets called and returns. The CPU hardware that tracks return addresses handles dispatch almost for free. Research shows this is 19–37% faster than direct threading.

Superinstructions

Some instruction sequences appear together so often that dispatching them individually is wasteful. COMPARE followed by JUMP_IF_TRUE is a classic example — virtually every if statement and loop generates this pair. A superinstruction fuses them into one: COMPARE_AND_JUMP. One dispatch instead of two. One trip through the loop instead of two.

The analogy: you don't spell out "do not" every time you speak — you say "don't." Same meaning, fewer steps. Superinstructions are contractions for bytecode.

UBI does this at load time. When a .ubi file is loaded into memory, the interpreter scans the bytecode, identifies common patterns, and rewrites them into fused superinstructions before execution starts. The program runs faster without the compiler having to know anything about which fusions exist — and new superinstructions can be added to the runtime without changing the bytecode format.

The GIL (and Why It Matters)

GIL stands for Global Interpreter Lock. CPython has one. UBI does not.

Here's what it is: a lock that says "only one thread can execute Python bytecode at a time." Even if your machine has 16 CPU cores, CPython uses exactly 1 of them for bytecode execution. If you write a multi-threaded Python program expecting it to run in parallel, it doesn't. The threads take turns, one at a time, passing the lock back and forth.

Why does it exist? CPython manages memory with reference counting — every object has a counter tracking how many things point to it, and when the counter hits zero, the object is freed. Reference counting is simple and predictable, but it's not thread-safe. Two threads incrementing the same counter at the same time can corrupt it. The GIL is the simplest possible fix: don't let two threads run at the same time, and the problem disappears.

UBI avoids the GIL because each of its memory management modes was designed to be thread-aware from the start. Reference counting uses atomic operations (hardware-guaranteed single-step increments that can't be corrupted by concurrent access). The garbage collector can run concurrently with program threads. There's no single lock because there doesn't need to be one — the thread safety is built into each individual operation rather than bolted on at the top.

Did you know? The word "bytecode" comes from the fact that each instruction's opcode fits in one byte (8 bits), giving a maximum of 256 different instructions. UBI uses exactly this: opcodes 0x00 through 0xFF.
