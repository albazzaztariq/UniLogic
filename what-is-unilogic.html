What is UniLogic?

What is this?

An open-source programming language and research project. UL (UniLogic) is a single language that compiles to C, Python, JavaScript, LLVM IR, and WebAssembly from the same source file. It is statically typed by default with an optional dynamic mode, has configurable runtime behavior per file, and treats AI/ML optimization as a first-class language concern rather than a library afterthought.

The project is open-source, documented as it develops, and not a product for sale. It's a research effort exploring whether a single unified language can replace the tradeoffs that currently force developers to choose between performance, portability, safety, and ecosystem access. Collaboration and feedback from anyone who finds this worth thinking about is welcome.

Why does this exist?

Every programming language makes tradeoff decisions that you inherit when you choose it. Python gives you an enormous ecosystem and fast development but is 10–100x slower than C, has a GIL that prevents real parallelism, and offers no path to native binaries. C gives you maximum performance and hardware access but has no memory safety, no package ecosystem comparable to pip, and alienates practitioners who aren't systems programmers. JavaScript runs everywhere but has no integer type, no static types without bolting on TypeScript, and no way to write low-level code. Java gives you type safety and threading but locks you into the JVM, GC you can't turn off, and startup times measured in seconds.

Each of these languages made decisions about memory management, type checking, safety guarantees, and concurrency at design time. Those decisions are fixed. You can escape them in narrow ways — Rust has unsafe, Python has Cython, JavaScript has WASM — but the escape hatches are limited, bolted on, and often require learning an entirely separate tool or language.

UL makes these decisions configurable. Memory model (GC, manual, refcount, arena), safety level (checked or unchecked), type system (static or dynamic), integer width (32, 64, or platform), concurrency model (threaded, parallel, async, cooperative) — all declared per file at compile time. The compiler enforces that files with conflicting settings don't interact unsafely.

How does it work?

The compiler (XPile) translates UL source into any of five targets. Compile to C or LLVM IR for native binaries with full hardware access. Compile to Python and your code runs anywhere a Python interpreter exists, with full access to the Python ecosystem (pip, numpy, pandas — all callable). Compile to JavaScript and it runs in any browser or Node.js environment. Compile to WebAssembly and it runs on any device with a browser, no install required.

Each target isn't a watered-down translation. The C output uses stdio.h, stdint.h, structs with inheritance via composition, and maps UL builtins to their C equivalents. The Python output uses dataclasses, native Python match/case, and idiomatic Python patterns. The JS output uses factory functions, strict equality, and console.log. You get the ecosystem of each target language with zero loss and everything gained from writing in a single syntax.

The standard library has one implementation that works across all runtime configurations. A linked list under GC doesn't need a destructor. Under manual memory it does. Under refcount it needs weak reference support. The standard library handles this — the developer declares what they want and the compiler selects the correct implementation path.

What about AI and ML?

AI needs a language. Python has the tooling (PyTorch, TensorFlow, Hugging Face) but is the wrong language for performance-critical computation. C and CUDA have the speed but alienate ML practitioners and require specialist knowledge to use effectively. The result is that the AI ecosystem is built on Python calling C calling CUDA — three languages, three build systems, three sets of abstractions leaking into each other.

The optimizations that matter for AI workloads are currently either clunky library operations or simply unavailable at the language level:

  • Mixed precision requires .half() calls scattered manually across an entire codebase
  • Quantization is a post-training library operation, not a type
  • Kernel fusion depends on whether torch.compile happens to recognize the pattern
  • Custom CUDA kernels require writing GPU programs in a separate language
  • Sparsity requires converting tensors to sparse formats and using separate library functions
  • Prefetch requires hardware-specific compiler intrinsics placed manually
  • Memory layout (struct-of-arrays vs array-of-structs) must be managed by hand

UL treats these as language-level concerns. Mixed precision is a compile-time setting — declare it once and the compiler applies FP16 computation with FP32 accumulation. Quantization is a type (int8 is first-class, not a library conversion). Sparsity is a type-level construct. Prefetch is a language annotation where the compiler calculates the optimal distance per target architecture. Memory layout is controlled via DR settings. FMA patterns are recognized and fused automatically.

What else?

  • Formal memory safety via SMT solver — the same mathematics used by Airbus and automotive manufacturers, built into the compiler
  • Error handling with Result types, ? propagation, and exit() — you choose the model per function
  • Match statement with pattern matching across all compilation targets
  • Type inheritance with structs and fields
  • Foreign function interface for calling C libraries directly
  • AST normalization at four levels for code simplification
  • VS Code extension with diagnostics, hover, autocomplete, signature help, go-to-definition, AI-powered code walkthroughs, and dependency graph visualization
  • Interactive browser REPL that runs entirely client-side with zero dependencies
  • DR conflict detection across files at compile time before anything runs
  • Bytecode VM specification (register-based, 32-bit fixed-width, tiered ISA)

The syntax borrows from what already exists across C, Python, Java, and JavaScript. The goal is to unify and minimize, not to invent. Verbosity is chosen over cleverness. Readability over elegance. Practicality over ideology.

The full design rationale and conceptual basis is documented in the LANG Project document.
