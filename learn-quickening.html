Optimizing Without a JIT

Quickening, inline caching, NaN boxing, and kill bits — the tricks that make interpreters fast without generating machine code.

The Problem — Type Checking on Every Operation

In a dynamic language, the expression a + b is ambiguous. If a and b are both integers, it means integer addition. If they're floats, float addition. If they're strings, concatenation. If one is a string and the other is an integer, it might be an error — or it might coerce one to the other, depending on the language. The interpreter cannot know which case it's in until it checks the types at runtime.

So every time the interpreter hits an ADD instruction, it has to: look at the type tag on a, look at the type tag on b, branch into the correct handler, and only then perform the actual operation. Even if a and b have been integers for the last 10,000 iterations of a loop, the interpreter still checks both types every single time through.

The type check often takes longer than the addition itself. An integer add is one CPU instruction. The type dispatch around it can be a dozen — loads, comparisons, branches. That overhead is the central performance problem of every dynamic-language interpreter, and the techniques on this page exist to eliminate it.

Quickening — Adaptive Specialization

The idea is simple: the first time an ADD instruction runs and sees two integers, the interpreter rewrites the opcode byte — in place, in memory — from ADD (generic, does type checking) to ADD_INT (specialized, assumes both operands are integers). The next time the interpreter reaches that instruction, it executes ADD_INT directly: load two integers, add them, store the result. No type check at all.

If the types ever change — someone passes a float where an integer used to be — the specialized instruction has a guard that catches this. The guard fails, the opcode reverts back to generic ADD, and the whole process starts over. The instruction re-specializes based on whatever types it sees next.

The rewrite is a single byte write in memory. The instruction doesn't move, doesn't resize, and nothing around it changes. It's the smallest possible modification. CPython added this technique in version 3.11 (2022), following PEP 659. UBI was designed with quickening from the start — its fixed 32-bit instruction format means the rewrite is always a single aligned byte write, with no bytecode surgery required.

Inline Caching

When your code calls obj.method(), the interpreter has to do a lot of work: determine the type of obj, look up "method" in that type's method table, verify it exists, get the function pointer, and finally call it. That lookup can walk through multiple levels of inheritance. It's expensive — and if this line is inside a loop, the interpreter does the full lookup every iteration.

Inline caching short-circuits this. The interpreter remembers the answer right next to the instruction: "last time I was here, obj was a Point and method was at address 0x7fff1234." Next time, it checks: is obj still a Point? If yes, call 0x7fff1234 directly. Skip the entire lookup chain.

The cache can hold more than one entry. A monomorphic cache has seen exactly one type — this is the fastest case. A polymorphic cache has seen 2–3 types and checks a short list. A megamorphic cache has seen too many types and gives up: it falls back to the full lookup every time. Most real-world call sites are monomorphic, which is why inline caching works so well in practice.

Did you know? V8 (Chrome's JavaScript engine) keeps inline caches with up to 4 entries per call site. If a 5th type shows up, V8 gives up on caching for that site and falls back to a full lookup every time. This is why JavaScript code that passes many different object shapes to the same function can be unexpectedly slow.

NaN Boxing — Packing Values Into 8 Bytes

A CPU register is 8 bytes (64 bits). If your interpreter's Value type is a struct containing a type tag, a payload, and some flags — say 40 bytes total — it doesn't fit in a register. Every operation has to read from and write to memory, which is orders of magnitude slower than working in registers.

NaN boxing exploits a quirk of IEEE 754 floating-point numbers. The spec defines a huge range of bit patterns — roughly 253 of them — that all represent "Not a Number" (NaN). In practice, only one or two of those patterns are ever used. The rest are wasted space. NaN boxing repurposes them: you can hide a pointer, an integer, a boolean, or a type tag inside those unused NaN bit patterns. The result is a Value type that fits in exactly 8 bytes — one register.

When the value is a float, you just store the float normally. When it's anything else, you encode it as a specially crafted NaN whose payload bits carry the actual data. The interpreter checks a few high bits to determine the type and extracts the value with a mask. Everything stays in registers. JavaScriptCore (Safari's JS engine) and LuaJIT both use NaN boxing. V8 uses a related technique called tagged pointers. The principle is the same: make every value fit in a single machine word so the CPU never has to go to memory for basic operations.

Kill Bits — Telling the Interpreter What's Dead

Consider the instruction x = a + b. After the addition, do we still need a and b? If this is the last time either variable is used, their register slots are wasted space — occupied but never read again. In a register-based interpreter with a fixed number of slots, that waste means other values get pushed to memory sooner.

Kill bits solve this. They are flags encoded directly in the instruction that say "this operand is used for the last time here." When the interpreter (or a JIT compiler, if one exists) sees the kill bit, it knows that register slot is immediately available for reuse. No liveness analysis at runtime, no scanning ahead — the compiler already did that work and baked the answer into the bytecode.

Think of it like returning a library book the moment you finish the last page, instead of keeping it on your desk until the due date. The book goes back into circulation sooner, and the library (the register file) can serve more readers with fewer shelves. UBI encodes kill bits at compile time, so the interpreter gets this information for free on every instruction.

Did you know? The term "quickening" comes from the Dalvik VM (Android's original Java runtime). The Dalvik team used it to mean "rewriting bytecode to a faster form at install time." CPython and UBI use it at runtime instead — the bytecode quickens itself as it runs.
