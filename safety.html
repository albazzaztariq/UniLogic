Memory Safety

Memory bugs are the single largest source of critical vulnerabilities in systems software. Use-after-free, buffer overreads, null dereferences, data races — they don't make your program crash politely. They silently corrupt data, expose secrets, or hand an attacker control of execution. Google has reported that ~70% of serious Chrome security bugs are memory safety issues. Microsoft: the same number for Windows CVEs.

Rust solved this with the borrow checker — a compile-time ownership system that tracks which code holds which memory and prevents you from breaking the rules. It works, but learning lifetimes, ownership, and borrowing as first-class concepts has a real cost, especially when you just want to write a fast, safe program and not a dissertation on memory regions.

UniLogic takes a different path: it proves memory safety mathematically. You write code. The compiler generates proof obligations and an SMT solver discharges them. If every obligation is satisfied, the program is proven safe for every possible input — not just the ones you tested. No lifetime annotations. No fighting the borrow checker. No runtime overhead.

Buffer overread

C
int arr[5] = {10,20,30,40,50};
printf("%d\n", arr[7]);
// reads garbage or corrupts
// compiles. no warning.
Rust
let arr = [10,20,30,40,50];
println!("{}", arr[7]);
// panics at runtime:
// index out of bounds
// caught — only when you run it
UL
array[int] arr = [10,20,30,40,50]
print arr[7]
// proof obligation: 0 ≤ 7 < 5
// 7 ≥ 5 is always false
// rejected at compile time
// no binary is produced
UL catches this at compile time. MemProof rejects it before the program compiles. There is no binary to ship and no test run needed.

Use-after-free

C
int *p = malloc(sizeof(int));
*p = 99;
free(p);
printf("%d\n", *p);
// silent undefined behavior
// may print 99, 0, or worse
Rust
let p = Box::new(99);
drop(p);
println!("{}", p);
// error[E0382]: use of moved value
// borrow checker: compile-time
// won't build
UL
// Base mode
int val = 99
int pointer p = address<val>
print jump<p>
// MemProof tracks \valid(p)
// freed or out-of-scope pointer
// → proof fails at compile time

A Further Option: Functional Correctness

Memory safety means your program doesn't corrupt memory. Functional correctness means your program does what it's supposed to do. These are different things. A memory-safe program can still sort a list in the wrong order, return the wrong value, or silently drop data. There's no crash, but it's wrong.

Rust's type system can prove a function won't crash, but it cannot prove the function returns the right answer. Tools like Prusti and Verus can do this for Rust — UL can do it too. You write a contract stating what the function promises, and the verifier proves every line of code upholds that promise for every possible input — not by testing, by proof. The difference is that in UL the toolchain is already there. Nothing extra to install or configure.

The bug that looks correct

C
int clamp(int v, int lo, int hi) {
  if (v < lo) return lo;
  if (v > hi) return lo; // bug
  return v;
}
// compiles. passes many tests.
// wrong when val > hi
Rust
fn clamp(v: i32, lo: i32, hi: i32)
       -> i32 {
  if v < lo { return lo; }
  if v > hi { return lo; } // bug
  v
}
// compiles. no warnings.
// Rust cannot see this
UL
@requires
  lo <= hi,
  lo <= result and result <= hi,
  v < lo and result == lo, v > hi and result == hi
end requires
function clamp(int v, int lo, int hi) returns int
  if v < lo
    return lo
  else if v > hi
    return lo  // bug
  end if
  return v
end function
// MemProof fails: when v > hi,
// result is lo but contract
// requires result == hi
// rejected at compile time

The contract is three lines. It fully specifies what clamp must do. The verifier doesn't need to know where the bug is — it tries to prove the contract holds and fails. The developer finds the bug before a binary exists. This is all optional. They're features if you want them, but not features you have to use.

What you can put in a contract

AnnotationMeaning
@requiresPrecondition — what must be true before the function runs
@ensuresPostcondition — what must be true when it returns. result refers to the return value.
@invariantLoop invariant — a condition that holds at the start of every iteration
@assignsFrame condition — exactly which memory locations this function is allowed to modify
Contracts are optional. Without any annotations, UL still automatically verifies basic memory safety — valid pointers, in-bounds array access, no use-after-free. Contracts let you go further and prove what the code does, not just that it doesn't crash.

How It Works

The verification engine is MemProof — a static formal verifier built into the UniLogic toolchain. Compiling with --memproof runs your source through a pipeline that ends in a machine-checkable mathematical proof: the program cannot cause a memory safety violation for any valid input.

The idea in plain terms

Your code goes in. First, the tool checks that it follows certain rules — like no unbounded loops — that make it possible to analyze completely. It then builds a picture of every value every variable could ever hold, no matter what input you give it.

For every place that could go wrong — every array read, every pointer follow — it generates one question: is there any input that breaks this? An algorithm then tries to find an input that does. If it can't find one after checking every possible case, that's the proof. If it does find one, it hands you the exact input that causes the problem. That's your bug.

If everything passes, the proof gets packaged up with some metadata — the hash of your source code, the version of the tool that ran it, the result of every individual check — and published. The actual proof material goes to distributed storage. A small summary record goes onto a blockchain permanently. Anyone, anywhere, at any time can pull that record, retrieve the proof, and confirm it themselves without trusting you or anyone else. The signing process is explained in more depth below.

The three outcomes

OutcomeMeaningShips?
PROVEDEvery check passed with formal proofYes
UNKNOWN — ANNOTATABLEA loop invariant or precondition is missing; developer can supply it and resubmitNo
UNKNOWN — STRUCTURALProgram uses a construct MemProof cannot handle automatically (e.g. non-linear index arithmetic)No
No annotation needed for basic memory safety. MemProof automatically generates the memory validity obligations — valid pointers, in-bounds array access, no use-after-free — without any annotation. The @requires and @ensures contracts are for proving what the code does, on top of that baseline.

MemProof & MemResistant

UniLogic has two safety modes for different stages of a project's life.

MemProof — mathematically proven

Full formal verification. The source is run through the eight-stage MemProof pipeline — safe subset enforcement, SSA IR construction, abstract interpretation, separation logic and shape analysis, verification condition generation, SMT solving via CVC5, LFSC proof witnesses, and certificate publication. Every proof obligation is discharged by the solver and confirmed by an independent kernel. The result is a proof certificate — a machine-checkable record that this program, at this exact version of the code, is proven memory safe for all possible valid inputs. This is the mode for deployment, certification, and anywhere a guarantee stronger than testing is required.

ul program.ul --memproof       // MemProof build + certificate
Slower to build, stronger to ship. Verification time depends on function complexity and whether the @safe subset is used. An incremental proof cache means only changed functions are reproved — most sessions re-verify a handful of functions, not the whole codebase. Run MemProof before release.

MemResistant — hardened, not proven

If you don't want to go through formal proof, you don't have to. MemResistant is the still-optional alternative. It doesn't prove, but it gives confidence. The binary is built with a full stack of runtime safety detectors that catch unsafe behavior the moment it occurs, aborting with a precise diagnostic instead of silently corrupting memory.

DetectorCatches
AddressSanitizer (ASan)Use-after-free, heap overflow, stack overflow, use-after-return
UndefinedBehaviorSanitizer (UBSan)Signed integer overflow, null pointer dereference, misaligned access, bad shifts
ThreadSanitizer (TSan)Data races, lock-order violations, and other concurrency bugs
CBMCBounded model checking — explores all execution paths up to a configurable loop depth
Bounds checkerEvery array access checked against its declared size at runtime
ul program.ul --fullsafety    // MemResistant build
Use MemResistant during development. Fast to build, immediate diagnostics on any unsafe behavior. You get a precise crash with a clear message instead of a silent corrupt state. It's the right tool if you don't want to deal with proofing or it's unnecessary.

Digital Signing

A proof is only as trustworthy as the chain of custody behind it. Once a program is proven safe, UniLogic makes that proof public and lets anyone in the world verify it independently — without trusting you, without trusting any authority, without re-running the verification.

Hashing

A hash is a mathematical function that converts any input into a fixed-length value. You can feed it a single character or an entire codebase, and it always produces exactly 64 hexadecimal characters. It only goes one way: the hash can be computed from the input, but the input cannot be reconstructed from the hash. Change even one character, and the hash changes completely and unpredictably.

UniLogic uses SHA-256, the same algorithm used across TLS certificates, Bitcoin, code signing, and modern encryption infrastructure. No two different inputs have ever been found to produce the same SHA-256 hash. That property is called collision resistance, and it is what makes the hash a trustworthy fingerprint.

IPFS and Arweave — content-addressed storage

The proof witnesses produced by MemProof are large. They go to IPFS (InterPlanetary File System), a distributed storage network where the address of a file is derived from its contents — store your proof, get a hash back, and that hash is the permanent address of those exact bytes on any IPFS node anywhere in the world. For permanent storage, witnesses are also pinned to Arweave, which guarantees perpetual availability.

The MemProof verifier source — the Coq implementation that defines what it means for a program to be proven safe — is also on IPFS. Its content ID (CID) is recorded in every certificate. Anyone can retrieve it, run coqc against the Coq proof, and confirm that the verifier itself is formally sound — not just that it produced a result.

Blockchain — permanent public record

The certificate JSON — roughly 1KB — goes to a smart contract on the blockchain. The certificate contains: the program's SHA-256 hash, the verifier's IPFS CID, the result of each verification condition (location, kind, status), and the hashes of the LFSC proof witnesses. Smart contracts are programs that run on a public blockchain — they cannot be modified, deleted, or tampered with after deployment. Anyone can read them. Recording your certificate on-chain creates a permanent, timestamped, public record: at this exact moment, this exact program was proven safe by this exact version of the verifier.

Independent verification

To verify a piece of UL code you received, you don't run MemProof again. You do this:

  1. Retrieve the MemProof Coq source from the certificate's verifier CID on IPFS
  2. Check the Coq proof with coqc — confirms MemProof is formally sound
  3. Look up the certificate on-chain by the program's hash
  4. Retrieve the LFSC proof witnesses from IPFS using the hashes in the certificate
  5. Run the LFSC kernel against each witness — confirms every VC was actually proved
  6. Confirm the program hash in the certificate matches what you have

No trust in the UniLogic team, the MemProof team, or any oracle is required. The math and the public record do it.

MemResistant runs are recorded too. The on-chain record distinguishes between a MemResistant certification and a MemProof certification. Anyone verifying your program can see exactly which level of guarantee it carries. 
// One-time: prove, publish, certify
ul certify program.ul

// Anyone, anytime: verify against the on-chain record
ul verify  program.ul
ul certify and ul verify are in development. The pipeline is designed and the tooling is being built.
