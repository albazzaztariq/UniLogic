Memory Safety

Memory bugs are the single largest source of critical vulnerabilities in systems software. Use-after-free, buffer overreads, null dereferences, data races — they don't make your program crash politely. They silently corrupt data, expose secrets, or hand an attacker control of execution. Google has reported that ~70% of serious Chrome security bugs are memory safety issues. Microsoft: the same number for Windows CVEs.

Rust solved this with the borrow checker — a compile-time ownership system that tracks which code holds which memory and prevents you from breaking the rules. It works. But learning lifetimes, ownership, and borrowing as first-class concepts has a real cost, especially when you just want to write a fast, safe program and not a dissertation on memory regions.

Pragma takes a different path: it proves memory safety mathematically. You write code. The compiler generates proof obligations. An SMT solver discharges them. If every obligation is satisfied, the program is proven safe for every possible input — not just the ones you tested. No lifetime annotations. No fighting the borrow checker. No runtime overhead.

Buffer overread

C
int arr[5] = {10,20,30,40,50};
printf("%d\n", arr[7]);
// reads garbage or corrupts
// compiles. no warning.
Rust
let arr = [10,20,30,40,50];
println!("{}", arr[7]);
// panics at runtime:
// index out of bounds
// caught — only when you run it
Pragma
array[int] arr = [10,20,30,40,50]
print arr[7]
// proof obligation: 0 ≤ 7 < 5
// 7 ≥ 5 is always false
// rejected at compile time
// no binary is produced
Pragma catches this earlier than Rust. Rust's bounds check is a runtime panic. Pragma's WP proof rejects it before the program compiles. There is no binary to ship. No test run needed.

Use-after-free

C
int *p = malloc(sizeof(int));
*p = 99;
free(p);
printf("%d\n", *p);
// silent undefined behavior
// may print 99, 0, or worse
Rust
let p = Box::new(99);
drop(p);
println!("{}", p);
// error[E0382]: use of moved value
// borrow checker: compile-time
// won't build
Pragma
// Base Pragma
int val = 99
int pointer p = address<val>
print jump<p>
// WP tracks \valid(p) through scope
// freed or out-of-scope pointer
// → proof fails at compile time

Functional Correctness

Memory safety means your program doesn't corrupt memory. Functional correctness means your program does what it's supposed to do. These are different things. A memory-safe program can still sort a list in the wrong order, return the wrong value, or silently drop data. No crash. Just wrong.

This is where Pragma goes further than Rust. Rust's type system can prove a function won't crash. It cannot prove the function returns the right answer. Pragma can. You write a contract stating what the function promises, and the verifier proves every line of code upholds that promise for every possible input — not by testing, by proof.

The bug that looks correct

C
int clamp(int v, int lo, int hi) {
  if (v < lo) return lo;
  if (v > hi) return lo; // bug
  return v;
}
// compiles. passes many tests.
// wrong when val > hi
Rust
fn clamp(v: i32, lo: i32, hi: i32)
       -> i32 {
  if v < lo { return lo; }
  if v > hi { return lo; } // bug
  v
}
// compiles. no warnings.
// Rust cannot see this
Pragma
@requires lo <= hi
@ensures  lo <= result and result <= hi
@ensures  v < lo  -> result == lo
@ensures  v > hi  -> result == hi
function clamp(int v, int lo, int hi) returns int
  if v < lo
    return lo
  end if
  if v > hi
    return lo  // bug
  end if
  return v
end function
// MemProof fails: when v > hi,
// result is lo but contract
// requires result == hi
// rejected at compile time

The contract is four lines. It fully specifies what clamp must do. The verifier doesn't need to know where the bug is — it tries to prove the contract holds and fails. The developer finds the bug before a binary exists.

What you can put in a contract

AnnotationMeaning
@requiresPrecondition — what must be true before the function runs
@ensuresPostcondition — what must be true when it returns. result refers to the return value.
@invariantLoop invariant — a condition that holds at the start of every iteration
@assignsFrame condition — exactly which memory locations this function is allowed to modify
Contracts are optional. Without any annotations, Pragma still automatically verifies basic memory safety — valid pointers, in-bounds array access, no use-after-free. Contracts let you go further and prove what the code does, not just that it doesn't crash.

How It Works

The verification engine is Frama-C with the WP (Weakest Precondition) plugin. Frama-C is an industrial-strength static analysis framework developed by CEA (French Alternative Energies and Atomic Energy Commission) and Inria. It is used in aerospace, automotive, and nuclear software certification. The WP plugin implements deductive verification — the kind that produces actual mathematical proofs, not just test coverage.

The idea in plain terms

Say you want to prove a combination lock only opens for the right code. You don't try every combination — that would take years. Instead you reason about the mechanism: given how the tumblers work, show that a wrong code cannot align all three. The proof covers every possible combination at once, by logic, not by exhaustion.

WP does the same thing with code. Starting from your postcondition — what the function must guarantee at the end — it walks backwards through every line, asking: what has to be true here for that final guarantee to hold? This generates a set of conditions called verification conditions. Each one is a precise mathematical statement. They get handed to an SMT solver — software that is very good at deciding whether a mathematical claim is always true, or whether it can be broken. If the solver confirms every condition, the proof is complete. If any condition fails, it produces a concrete counterexample: here is the exact input that breaks your contract.

SMT solvers are the same technology used to verify processor designs at Intel and AMD, ensure safety properties in aircraft control systems, and audit financial contracts. Applying them to everyday compiled programs is what's new.

No annotation needed for memory safety. Frama-C WP automatically generates the memory validity obligations — \valid(ptr), array bounds, no overlap — without you writing a single annotation. The @ensures and @requires contracts are for proving what the code does, on top of that baseline.

MemProof & MemResistant

Pragma has two safety modes for different stages of a project's life.

MemResistant — hardened, not proven

MemResistant doesn't prove. It arms. The binary is built with a full stack of runtime safety detectors that catch unsafe behavior the moment it occurs, aborting with a precise diagnostic instead of silently corrupting memory.

DetectorCatches
AddressSanitizer (ASan)Use-after-free, heap overflow, stack overflow, use-after-return
UndefinedBehaviorSanitizer (UBSan)Signed integer overflow, null pointer dereference, misaligned access, bad shifts
CBMCBounded model checking — explores all execution paths up to a configurable loop depth
Bounds checkerEvery array access checked against its declared size at runtime
pragma program.run --fullsafety    // MemResistant build
Use MemResistant during development. Fast to build, immediate diagnostics on any unsafe behavior. You get a precise crash with a clear message instead of a silent corrupt state. The right tool while iterating.

MemProof — mathematically proven

Full Frama-C WP verification. Every proof obligation is discharged by an SMT solver. The result is a proof certificate — a machine-checkable record that this function, at this exact version of the code, is proven safe and correct for all possible inputs. This is the mode for deployment, certification, and anywhere a guarantee stronger than testing is required.

pragma program.run --memproof       // MemProof build + certificate
Slower to build, stronger to ship. WP verification ranges from seconds to minutes depending on function complexity. An incremental proof cache means only changed functions are reproved — most sessions re-verify a handful of functions, not the whole codebase. Run MemProof before release.

Digital Signing

A proof is only as trustworthy as the chain of custody behind it. Once a program is proven safe, Pragma makes that proof public and lets anyone in the world verify it — without trusting you, without trusting a central authority.

Hashing

A hash is a mathematical function that converts any input into a fixed-length value — you can feed it a single character or an entire codebase, and it always produces exactly 64 hexadecimal characters. It only goes one way: the hash can be computed from the input, but the input cannot be reconstructed from the hash. Change even one character, and the hash changes completely and unpredictably.

Pragma uses SHA-256, the same algorithm used across TLS certificates, Bitcoin, code signing, and modern encryption infrastructure. No two different inputs have ever been found to produce the same SHA-256 hash. That property is called collision resistance, and it is what makes the hash a trustworthy fingerprint.

IPFS — content-addressed storage

The proof certificate produced by MemProof is stored on IPFS (InterPlanetary File System), a distributed storage network. IPFS is content-addressed: the address of a file is derived from its contents. Store your certificate, get a hash back. That hash is the permanent address of those exact bytes — on any IPFS node, anywhere in the world, forever. Change one byte and you get a completely different address. The original is immutable.

Ethereum smart contract — public timestamp

The IPFS hash gets written to an Ethereum smart contract. Smart contracts are programs that run on a public blockchain — they cannot be modified, deleted, or tampered with after deployment. Anyone can read them. Writing your IPFS hash to the contract creates a permanent, timestamped, public record: at this exact moment, this exact proof existed and was submitted by this address.

Verification

To verify a piece of Pragma code you received, you compile it yourself and run MemProof. You get a hash. You compare it to what is recorded on-chain. If they match, you are looking at the exact code that was formally proven — byte for byte, character for character. If they don't match, the code has been modified since it was certified.

You don't need to trust the developer. You don't need to trust a registry. The math and the chain do it for you.

MemResistant runs are recorded too. The on-chain record distinguishes between a MemResistant certification and a MemProof certification. Anyone verifying your program can see exactly which level of guarantee it carries. 
// One-time: prove, publish, certify
pragma certify program.run

// Anyone, anytime: verify against the on-chain record
pragma verify  program.run
pragma certify and pragma verify are in development. The pipeline is designed. The tooling is coming.
