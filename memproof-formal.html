UniLogic MemProof — Formal Specifications

Version 0.1

Industry context. The formal methods described in this document — Hoare logic, weakest precondition calculus, SMT-based discharge, abstract interpretation — are the same techniques used across safety-critical industries. Airbus uses Frama-C and SPARK/Ada for flight control software under DO-178C. Automotive manufacturers apply these methods under ISO 26262 and MISRA-C. NASA, the FDA (for medical device software), and defense contractors rely on the same mathematical foundations to prove that software cannot fail in ways testing alone cannot rule out. MemProof applies these established methods to UL.

1. Goals and Scope

MemProof is UniLogic's static formal verification layer. Its purpose is to prove, prior to deployment, that a UL program is memory safe — meaning the program cannot, for any valid input within its declared input contracts, exhibit:

  • Buffer overflows or out-of-bounds array access
  • Null pointer dereferences
  • Integer overflow (for types where overflow is undefined)
  • Use-after-free
  • Invalid pointer arithmetic

MemProof operates on the UL AST before code generation. It does not verify the generated C — it verifies the UL source against UL's formal semantics. This distinction matters: a bug in UL's C codegen cannot produce a false positive MemProof certificate.

MemProof produces one of three outcomes:

  • PROVED — a formal proof certificate is issued, published to IPFS, and recorded on chain.
  • UNKNOWN — ANNOTATABLE — verification failed due to insufficient invariant information; the developer can supply annotations to resolve.
  • UNKNOWN — STRUCTURAL — verification failed due to a language feature MemProof cannot handle automatically; requires manual review or program restructuring.

For safety-critical deployment, UNKNOWN is treated as UNSAFE. The program does not ship.

2. Memory Model

2.1 Formal Definition

UniLogic's memory model is adapted from CompCert's block-offset model. Memory is not modelled as a flat address space. Instead:

A block \(b\) is an abstract allocation unit with a unique identifier \(b \in \mathsf{Block}\). Each block has a size \(|b| \in \mathbb{N}\).

Memory is a partial function:

\[ M : (\mathsf{Block} \times \mathsf{Offset}) \to \mathsf{Value} \cup \{\bot\} \]

where \(\mathsf{Offset} \in \mathbb{Z}\), \(\mathsf{Value}\) is the set of all typed values (integers, booleans, pointers), and \(\bot\) denotes an unallocated or inaccessible cell.

A pointer in this model is a pair \((b, \delta)\) where \(b\) is a block identifier and \(\delta \in \mathbb{Z}\) is an offset into that block. A pointer is valid iff \(0 \leq \delta < |b|\).

This gives precise definitions of:

  • Null pointer: the distinguished pair \((\mathsf{null\_block},\, 0)\) where \(\mathsf{null\_block}\) is reserved and never allocated.
  • Out-of-bounds access: any access at \((b, \delta)\) where \(\delta < 0\) or \(\delta \geq |b|\).
  • Dangling pointer: a pointer \((b, \delta)\) where \(b\) has been freed; \(M(b, \cdot) = \bot\).

2.2 Memory Operations

Four primitive operations define the memory model:

alloc(M, n)       → (M', b)    allocate n bytes; return updated memory and fresh block b
free(M, b)        → M'         deallocate block b; M'(b, ·) = ⊥
load(M, b, δ)     → v          read value at (b, δ); defined only if M(b, δ) ≠ ⊥
store(M, b, δ, v) → M'         write value v at (b, δ)

Preconditions (as proof obligations):

  • \(\mathsf{load}(M, b, \delta)\) requires \(M(b, \delta) \neq \bot\) and \(0 \leq \delta < |b|\)
  • \(\mathsf{store}(M, b, \delta, v)\) requires the same plus write permission
  • \(\mathsf{free}(M, b)\) requires \(b\) was previously allocated and not yet freed

These preconditions become the verification conditions (VCs) that MemProof must prove hold for all valid executions.

2.3 Permission Model

Each cell \((b, \delta)\) carries a permission tag from the lattice:

\[ \mathsf{None} \sqsubseteq \mathsf{Read} \sqsubseteq \mathsf{ReadWrite} \]

\(\mathsf{None}\) — cell is unallocated or freed. \(\mathsf{Read}\) — cell may be loaded but not stored. \(\mathsf{ReadWrite}\) — cell may be loaded and stored. Permissions are tracked through the abstract interpretation pass and used by the separation logic heap predicates.

3. Separation Logic

Separation logic extends Hoare logic with predicates that reason about disjoint regions of the heap. This makes it possible to state and verify properties about pointer-heavy code without global aliasing analysis.

3.1 Heap Predicates

emp — the heap is empty (no allocated cells).

Points-to predicate:

\[ (b, \delta) \mapsto v \]

Asserts that cell \((b, \delta)\) is allocated, has permission ReadWrite, and contains value \(v\).

Array predicate:

\[ \mathsf{array}(b, \delta, n, \mathit{vs}) \]

Asserts that the \(n\) cells \((b, \delta), (b, \delta{+}1), \ldots, (b, \delta{+}n{-}1)\) are allocated and contain values \(\mathit{vs}[0], \ldots, \mathit{vs}[n{-}1]\). Defined inductively:

\[ \mathsf{array}(b, \delta, 0, []) \;\equiv\; \mathsf{emp} \]

\[ \mathsf{array}(b, \delta, n, v{::}\mathit{vs}) \;\equiv\; (b, \delta) \mapsto v \;\ast\; \mathsf{array}(b, \delta{+}1, n{-}1, \mathit{vs}) \]

3.2 Separating Conjunction

The key operator in separation logic is separating conjunction \(P \ast Q\):

\[ (M, h) \models P \ast Q \;\iff\; \exists\, h_1, h_2.\; h = h_1 \uplus h_2 \;\wedge\; (M, h_1) \models P \;\wedge\; (M, h_2) \models Q \]

where \(\uplus\) denotes disjoint union of heap segments. \(P \ast Q\) asserts that \(P\) and \(Q\) hold on disjoint parts of the heap. This eliminates the aliasing problem: if \(P\) describes a buffer and \(Q\) describes a pointer, and they are joined by \(\ast\), they provably do not overlap.

3.3 Magic Wand

The magic wand (separating implication) \(P \mathrel{-\!\!\ast} Q\):

\[ (M, h) \models P \mathrel{-\!\!\ast} Q \;\iff\; \forall h'.\; h \uplus h' \text{ defined} \;\Rightarrow\; (M, h') \models P \;\Rightarrow\; (M, h \uplus h') \models Q \]

Used primarily in function postconditions to describe what is returned to the caller's heap context.

3.4 Hoare Triples

A Hoare triple in separation logic:

\[ \{P\}\; C\; \{Q\} \]

asserts: if \(C\) executes in a heap satisfying \(P\), and \(C\) terminates, then the resulting heap satisfies \(Q\). For an array access \(x = \mathtt{arr}[i]\):

\[ \{\,\mathsf{array}(b, 0, n, \mathit{vs}) \ast i \mapsto k \ast 0 \leq k < n\,\} \;\; x = \mathtt{arr}[i] \;\; \{\,\mathsf{array}(b, 0, n, \mathit{vs}) \ast i \mapsto k \ast x = \mathit{vs}[k]\,\} \]

The precondition \(0 \leq k < n\) is the bounds check. MemProof must prove this holds.

3.5 The Frame Rule

The frame rule is what makes separation logic compositional:

\[ \frac{\{P\}\; C\; \{Q\}}{\{P \ast R\}\; C\; \{Q \ast R\}} \quad \bigl(\mathsf{modifies}(C) \cap \mathsf{fv}(R) = \emptyset\bigr) \]

If \(C\) is verified with pre \(P\) and post \(Q\), it can be lifted to any context \(R\) that \(C\) does not modify. This allows local reasoning: verify each function in isolation, compose globally.

3.6 Key Inference Rules Used by VCG

Load:

\[ \{(b, \delta) \mapsto v \ast R\} \;\; x = \mathsf{load}(b, \delta) \;\; \{(b, \delta) \mapsto v \ast x = v \ast R\} \]

Store:

\[ \{(b, \delta) \mapsto \_ \ast R\} \;\; \mathsf{store}(b, \delta, v) \;\; \{(b, \delta) \mapsto v \ast R\} \]

Alloc:

\[ \{\mathsf{emp}\} \;\; b = \mathsf{alloc}(n) \;\; \{\mathsf{block}(b, 0, n)\} \]

Free:

\[ \{\mathsf{block}(b, 0, n)\} \;\; \mathsf{free}(b) \;\; \{\mathsf{emp}\} \]

4. Shape Analysis

Separation logic alone cannot automatically reason about unbounded heap structures (linked lists, trees, dynamic arrays). Shape analysis provides inductive predicate definitions that describe the shape of such structures, enabling automated reasoning.

4.1 Inductive Predicates

A singly-linked list rooted at pointer \(p\) with values \(\mathit{vs}\):

\[ \mathsf{list}(p, \mathit{vs}) \;\equiv\; (p = \mathsf{null} \wedge \mathit{vs} = []) \;\vee\; \exists\, v, q, \mathit{vs}'.\; p \mapsto \{v, q\} \ast \mathsf{list}(q, \mathit{vs}') \wedge \mathit{vs} = v :: \mathit{vs}' \]

An array of length \(n\) at block \(b\):

\[ \mathsf{arr}(b, n, \mathit{vs}) \;\equiv\; \mathsf{array}(b, 0, n, \mathit{vs}) \wedge |\mathit{vs}| = n \]

4.2 Bi-abduction

Bi-abduction solves: given a program state \(S\) and a required footprint \(F\), find:

  • Anti-frame \(A\) such that \(S \ast A \models F\) (what needs to be added to make the precondition hold)
  • Frame \(R\) such that \(F \models S \ast R\) (what is left over)

This is used by the VCG to automatically infer function preconditions and postconditions from the function body, without requiring full manual specification.

4.3 Region-based Heap Abstraction

For scalability, the heap is abstracted into regions — sets of cells treated uniformly. Each region carries a base pointer, a size, a permission tag, and a summary predicate (e.g., array or list). Region merging and splitting rules govern how regions are updated at array accesses, pointer arithmetic, and function calls.

5. Abstract Interpretation

Abstract interpretation provides a framework for computing sound over-approximations of program behaviour — covering all possible executions without executing the program.

5.1 Lattice Theory

An abstract domain is a complete lattice \((A, \sqsubseteq, \sqcup, \sqcap, \bot, \top)\) where:

  • \(\sqsubseteq\) is the partial order (more precise \(\sqsubseteq\) less precise)
  • \(\sqcup\) is join (least upper bound — union of information)
  • \(\sqcap\) is meet (greatest lower bound — intersection)
  • \(\bot\) is the bottom element (unreachable / no information)
  • \(\top\) is the top element (all possible values / no information retained)

The concrete domain \((\wp(\mathsf{State}), \subseteq)\) is the powerset of all possible program states. The abstract domain approximates this.

5.2 Galois Connection

The relationship between concrete and abstract domains is formalised as a Galois connection:

\[ (\wp(\mathsf{State}),\, \subseteq) \;\underset{\gamma}{\overset{\alpha}{\rightleftharpoons}}\; (A,\, \sqsubseteq) \]

where \(\alpha : \wp(\mathsf{State}) \to A\) is the abstraction function and \(\gamma : A \to \wp(\mathsf{State})\) is the concretisation function. The Galois connection requires:

\[ \forall\, S \in \wp(\mathsf{State}),\, a \in A. \quad \alpha(S) \sqsubseteq a \;\iff\; S \subseteq \gamma(a) \]

This guarantees soundness: any concrete state reachable at a program point is represented by the abstract element at that point.

5.3 Abstract Domains Used

Interval domain: Each variable \(x\) is tracked as \([l, u]\) where \(l \leq x \leq u\). Transfer functions update intervals through arithmetic. Used for bounds checking: if \(i \in [0, n{-}1]\) and the array has length \(n\), access is provably safe.

Octagon domain: Tracks constraints of the form \(\pm x \pm y \leq c\). More precise than intervals for relational properties (e.g., \(i \leq j \leq n\)). Used when loop variables have relational bounds.

Polyhedra domain: Tracks linear inequalities over all variables simultaneously. Most precise but most expensive. Used selectively for critical safety regions.

5.4 Transfer Functions

Each program statement \(s\) has an abstract transfer function \(F_s^\sharp : A \to A\):

\[ F_{x := e}^\sharp(a) = a[x \mapsto \mathsf{eval}^\sharp(e, a)] \]

\[ F_{\mathsf{guard}(e)}^\sharp(a) = a \sqcap \mathsf{constraints}(e) \]

\[ F_{s_1;\, s_2}^\sharp(a) = F_{s_2}^\sharp(F_{s_1}^\sharp(a)) \]

5.5 Widening and Convergence

For loops, naive iteration may not converge. The widening operator \(\nabla : A \times A \to A\) forces convergence:

\[ a_0 = \bot \qquad a_{i+1} = a_i \;\nabla\; F_{\mathsf{loop}}^\sharp(a_i) \]

Widening terminates in a finite number of steps by construction. The result over-approximates all loop iterations — it may lose precision (reporting \(i \in [0, \infty)\) instead of \(i \in [0, n]\)) but is always sound.

Narrowing \(\triangle\) is then applied to recover precision:

\[ b_0 = a_{\mathsf{fixpoint}} \qquad b_{i+1} = b_i \;\triangle\; F_{\mathsf{loop}}^\sharp(b_i) \]

Narrowing tightens the over-approximation without losing soundness, recovering the exact bound when possible. For RTOS infinite control loops, widening produces \([0, \infty)\) for the loop counter, but narrowing with the loop guard condition \(i < \mathtt{SENSOR\_RANGE\_MAX}\) recovers \([0, \mathtt{SENSOR\_RANGE\_MAX})\), which is sufficient for bounds checking.

6. Verification Condition Generator (VCG)

The VCG is the central component of MemProof. It bridges the separation logic specification, the abstract interpretation results, and the SMT solver.

6.1 IR Construction

Before VCG, the UL AST is lowered to a typed SSA IR (Static Single Assignment). In SSA, each variable is assigned exactly once; control flow joins use \(\phi\)-functions: \(x_3 = \phi(x_1, x_2)\) at merge points; all loop variables have explicit induction structure. SSA simplifies the VCG because each use of a variable traces unambiguously to its definition.

6.2 Weakest Precondition Calculus

For a statement \(C\) and postcondition \(Q\), the weakest precondition \(\mathsf{wp}(C, Q)\) is the weakest condition such that \(\{\mathsf{wp}(C,Q)\}\; C\; \{Q\}\) holds.

\[ \mathsf{wp}(\mathsf{skip},\, Q) = Q \]

\[ \mathsf{wp}(x := e,\, Q) = Q[x \mapsto e] \]

\[ \mathsf{wp}(s_1;\, s_2,\, Q) = \mathsf{wp}(s_1,\, \mathsf{wp}(s_2,\, Q)) \]

\[ \mathsf{wp}(\mathsf{if}\; b\; \mathsf{then}\; s_1\; \mathsf{else}\; s_2,\, Q) = (b \Rightarrow \mathsf{wp}(s_1, Q)) \wedge (\neg b \Rightarrow \mathsf{wp}(s_2, Q)) \]

\[ \mathsf{wp}(\mathsf{load}(b, \delta),\, Q) = \mathsf{valid}(b, \delta) \wedge Q[x \mapsto M(b, \delta)] \]

\[ \mathsf{wp}(\mathsf{store}(b, \delta, v),\, Q) = \mathsf{valid}(b, \delta) \wedge \mathsf{writable}(b, \delta) \wedge Q[M \mapsto M[(b,\delta) \mapsto v]] \]

\[ \mathsf{wp}(\mathsf{free}(b),\, Q) = \mathsf{allocated}(b) \wedge Q[M \mapsto M \setminus b] \]

For loops, \(\mathsf{wp}\) requires a loop invariant \(I\):

\[ \mathsf{wp}(\mathsf{while}\; b\; \mathsf{do}\; s,\, Q) = I \;\wedge\; (I \wedge b \Rightarrow \mathsf{wp}(s, I)) \;\wedge\; (I \wedge \neg b \Rightarrow Q) \]

The VCG emits three obligations per loop: (1) \(I\) holds on entry; (2) \(I\) is maintained by the loop body; (3) \(\neg b \wedge I \Rightarrow Q\).

6.3 Memory Safety VCs

6.4 VC Annotation with Abstract State

Before emitting a VC to the SMT solver, the VCG enriches it with the abstract state computed by abstract interpretation at that program point. A raw VC:

\[ 0 \leq i \wedge i < \mathsf{arr\_len} \]

becomes, after enrichment:

Hypothesis (from abstract interpretation):  i ∈ [0, n-1]
Hypothesis (from abstract interpretation):  arr_len = n
VC:  0 ≤ i ∧ i < arr_len

The SMT solver can discharge this immediately from the hypotheses — the abstract interpretation result provides the numeric evidence, the SMT solver confirms the arithmetic.

7. SMT Encoding and Solving

7.1 Theories Used

MemProof uses the following SMT theories (all supported by CVC5):

  • QF_LIA — Quantifier-free linear integer arithmetic. Used for bounds checks, index bounds, overflow checks.
  • QF_BV — Bitvectors. Used for exact overflow checking on fixed-width integers (int8, int32, uint64, etc.)
  • QF_AX — Theory of arrays. Used for array read/write reasoning.
  • SL — CVC5's native separation logic theory. Used for heap predicate reasoning (points-to, separating conjunction). This avoids encoding separation logic into base theories, which is less efficient and less complete.

7.2 VC Format (SMT-LIB2)

(set-logic QF_LIA)
(declare-const i Int)
(declare-const arr_len Int)
(assert (>= i 0))          ; hypothesis from abstract interpretation
(assert (< i arr_len))     ; hypothesis from abstract interpretation
(assert (= arr_len 10))    ; hypothesis from constant propagation
; VC negation: try to find a counterexample
(assert (not (and (>= i 0) (< i arr_len))))
(check-sat)
; Expected: unsat — no counterexample exists → VC proved

If the solver returns unsat, the VC is proved. If sat, the solver produces a counterexample — a concrete input violating the property — which is reported to the developer as a bug. If unknown, the VC is unresolved and the outcome is UNKNOWN-STRUCTURAL.

7.3 CVC5 Native Separation Logic

For heap VCs, CVC5's SL theory is used directly:

(set-logic QF_BVLIA_SL)
(declare-heap (Int Int))
(assert (sep (pto p v) (pto q w)))
; p and q are disjoint heap locations
(assert (not (= p q)))
(check-sat)
; Expected: unsat — separating conjunction guarantees disjointness

7.4 Proof Witnesses (LFSC)

When CVC5 returns unsat, it produces an unsatisfiability proof in LFSC (Logical Framework with Side Conditions) format. This is the proof witness: a formal proof term that can be checked by the LFSC kernel independently of CVC5. The kernel is approximately 5,000 lines of auditable code. It has been itself formally verified. Its sole function is: given a proof term, confirm it is valid. It has no heuristics.

The proof witness for each VC is collected and bundled into the overall MemProof certificate:

{
  "program_hash":   "<sha256 of UL source>",
  "verifier_cid":   "<IPFS CID of MemProof version used>",
  "memory_model":   "block-offset-v1",
  "vcs": [
    {
      "location":     "main.ul:42",
      "kind":         "array_bounds",
      "status":       "proved",
      "witness_hash": "<sha256 of LFSC proof witness>"
    }
  ],
  "outcome":        "PROVED",
  "timestamp":      "<ISO 8601>",
  "assumptions":    [],
  "annotations_used": []
}

Proof witnesses are stored on IPFS (referenced by hash). The blockchain records only the certificate JSON. Anyone can retrieve the witnesses from IPFS and check them against the LFSC kernel.

8. Soundness

8.1 Soundness Theorem

MemProof is sound if: whenever it returns PROVED, the program is genuinely memory safe. Formally:

\[ \forall\, \mathit{prog},\, \mathit{cert}.\quad \mathsf{MemProof}(\mathit{prog}) = \mathsf{Proved}(\mathit{cert}) \;\Rightarrow\; \forall\, i \in \mathsf{valid\_inputs}(\mathit{prog}).\;\; \mathsf{exec}(\mathit{prog},\, i) \text{ does not exhibit a memory safety violation} \]

Soundness does not require completeness — MemProof may return UNKNOWN on some safe programs. It only requires that PROVED means genuinely safe.

8.2 Coq Formalization Strategy

To formally verify MemProof's soundness, the core components are implemented directly in Coq:

  1. Memory model — formalised as a Coq inductive type with operations and permission invariants proved as lemmas.
  2. Separation logic — the assertion language, satisfaction relation, and inference rules formalised. Soundness of each rule proved as a Coq theorem.
  3. Abstract interpretation framework — the Galois connection, transfer functions, widening/narrowing proved to be sound over-approximations.
  4. VCG — proved to generate only valid VCs: if the VCG emits \(\{P\}\; C\; \{Q\}\), then \(\{P\}\; C\; \{Q\}\) holds in the semantics.
  5. SMT encoding — proved that the SMT-LIB2 encoding of each VC is equisatisfiable with the semantic VC.

The top-level soundness theorem in Coq:

Theorem memproof_sound :
  forall prog : ULProgram,
  forall cert : Certificate,
  memproof prog = Proved cert ->
  memory_safe prog.

The Coq source is extracted to OCaml via Extraction. The extracted OCaml is what pragma.py calls as a subprocess. The Coq source is published on IPFS — its CID is recorded in every certificate, forming the auditable chain of trust.

9. UniLogic Safe Subset

Inspired by MISRA-C, the @safe decorator on a UL module or function enforces a restricted subset that makes verification tractable:

RestrictionReason
No dynamic allocation without static analysis proof of safetyPrevents unbounded heap growth; simplifies shape analysis
No unbounded recursionPrevents non-termination; loops must have loop variants
All loops must have a statically visible bound or @invariant annotationRequired for abstract interpretation convergence
No function pointers in safety regionsPrevents indirect call analysis failures
Integer types must be explicit width (int32, uint8, not bare int)Required for exact overflow checking via bitvectors
No type punning via union in safe regionsEliminates aliasing ambiguity
Pointer arithmetic only within declared boundsPrevents region escaping

Violation of safe subset rules is a compile error when @safe is present — not a verification failure. The compiler rejects the program before MemProof runs. This is how Airbus achieves near-complete automatic verification.

10.3 Independent Verification

Anyone can verify a certificate without trusting any party:

  1. Retrieve MemProof source from verifierCID on IPFS
  2. Check Coq proof with coqc — confirms MemProof is sound
  3. Retrieve certificate JSON from chain
  4. Retrieve LFSC witnesses from IPFS (referenced in certificate)
  5. Run LFSC kernel against witnesses — confirms each VC was proved
  6. Confirm program_hash matches the deployed program source

No trust in the UniLogic team, MemProof team, or any oracle is required.

11. Limitations and Known Incompleteness

MemProof is sound but not complete. Known cases where UNKNOWN is expected:

  • Non-linear arithmetic in array indices (e.g., arr[i * stride + offset] where both i and stride are variables) — QF_NIA is undecidable in general.
  • Complex recursion with non-trivial termination arguments — requires custom termination proofs.
  • Aliasing through function pointers — requires points-to analysis that may not converge.
  • External function calls without declared contracts — MemProof treats them as potentially modifying any memory.

In all UNKNOWN cases, MemProof reports the specific VC that could not be discharged, the program location, and whether developer annotation would resolve it.
