Pragma

A backend data movement layer for stateless sequential pipelines.
Sits above UniLogic — not a replacement for programming.

Pragma is part of the LANG Project.

What it is

Pragma is a declarative pipeline language for data movement. Each pipeline is a flat sequence of named actions — fetch, convert, save, notify, and others — each with key-value attributes. There is no control flow, no variables in the programming sense, and no business logic. The output of one action becomes the input to the next by name assignment. When a pipeline requires branching, conditionals, or computation, that work belongs in UniLogic. Pragma generates UniLogic code; changes to the generated code fold back up into the Pragma representation.

Syntax rules

All Pragma pipelines follow the same flat indented structure regardless of what actions are used.

action as header Each step opens with its action name followed by a colon. The action name is the only thing on that line.
key-value beneath Attributes are indented one level under the action header. Each attribute is key=value on its own line.
name= assigns output name=identifier captures the result of that step. Subsequent steps reference it by that identifier.
[a, b, c] — multi-value Square brackets with comma separation indicate a list of accepted or produced values.
[a or b or c] — fallback Square brackets with or separation indicate ordered fallback — try the first, fall back to the next on failure.
no nesting Actions do not nest inside each other. The pipeline is always a flat sequence. If nesting is needed, the logic belongs in UniLogic.

Code example — fetch, convert, save

The most common Pragma pattern: pull data from a source, transform its format, write it to a destination.

// Step 1 — fetch JSON from an API endpoint
fetch:
from=[http or file] source=https://api.example.com/records auth=api_auth format=json structure=openalex.map name=raw_records
// Step 2 — convert to xlsx for delivery
convert:
content=raw_records from=json to=xlsx name=report
// Step 3 — save to cloud storage
save:
to=cloud dest=reports/monthly.xlsx content=report format=xlsx mode=overwrite

Compilation

Pragma compiles to UniLogic. Each action in a pipeline generates the corresponding UL code — fetch becomes an HTTP call, convert becomes a format transformation function, save becomes a write operation. The generated UL is the authoritative output; Pragma is the high-level representation above it.

Changes made directly in the generated UL fold back up: the compiler reads the modified UL, reconstructs the pipeline attributes that produced it, and updates the Pragma source to match. The two representations stay in sync.

What Pragma covers — and what it doesn't

The boundary is stateless sequential pipelines. A step takes input, produces output, and does not depend on the results of steps other than those explicitly passed to it by name. If a pipeline step needs to examine its own previous output, branch on a condition, loop, maintain state, or perform computation beyond format transformation, it has left Pragma's domain.

The natural test: if you can describe the step as an action with a flat set of attributes and a named result, it belongs in Pragma. If you need an if, a loop, a variable that accumulates across steps, or any logic that refers to itself — write it in UniLogic and call it from the pipeline.

Pragma handles

  • Fetching data from a source
  • Converting between formats
  • Saving to a destination
  • Querying a database
  • Sending a notification
  • Scheduling a recurring action
  • Collecting user input
  • Displaying output

UniLogic handles

  • Conditional branching
  • Loops and iteration
  • Stateful accumulation
  • Business logic and computation
  • Error recovery with decisions
  • Anything that refers to itself
