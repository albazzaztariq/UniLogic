Dynamic Runtime — DR

Runtime Policy

Pragma exposes 13 runtime settings that cover the decisions most languages make once at the language level and never revisit. Each setting is declared at the top of a source file and enforced by the compiler for that file. There are no global defaults — what you write is what you get.

Most languages pick a memory model, a concurrency model, and an error strategy at design time. You inherit them when you choose the language, and working around them — if possible at all — requires unsafe blocks, external libraries, or runtime flags. Pragma treats these as per-file compile-time configuration. The standard library is implemented to work correctly under all combinations.

memory: gc // or: manual · refcount · arena
safety: checked // or: unchecked
concurrency: threads // or: async · cooperative
int_width: 32 // or: 64 · platform
float_model: ieee // or: fast
overflow: trap // or: wrap · saturate
error_model: exceptions // or: result · abort
types: strict // or: dynamic
allocator: system // or: custom · embedded
bounds_check:enabled // or: disabled
stack_size: default // or: N (in KB, e.g. 512)
exceptions: unchecked // or: checked
debug: symbols // or: stripped

memory

Controls how the program allocates and frees heap memory. This setting affects every allocation in the file. The standard library's collection types, strings, and data structures all work under any model.

memory gc  |  manual  |  refcount  |  arena

gc — Garbage collected. The runtime tracks live objects and reclaims unreachable memory automatically. Lowest programmer effort; some pause latency.

manual — Programmer manages allocation and deallocation. Use malloc, free. Full control, zero overhead, full responsibility.

refcount — Reference counting. Objects are freed when their reference count drops to zero. Deterministic timing; cycles require explicit handling.

arena — All allocations are made from a single region. The entire arena is freed at once. Common in request-handling servers and parsers.

default: gc

allocator

Selects the underlying allocator used for heap operations. Applies when memory is manual or gc.

allocator system  |  custom  |  embedded

system — Uses the platform allocator (malloc/free on POSIX, HeapAlloc on Windows).

custom — Binds to a user-provided allocator struct implementing the allocator interface. Useful for memory pools, instrumentation, or replacing jemalloc/mimalloc.

embedded — Statically-sized allocator with no dependency on the OS heap. For bare-metal and firmware targets with no dynamic memory subsystem.

default: system

overflow

Determines what happens when an integer arithmetic operation exceeds the type's range.

overflow trap  |  wrap  |  saturate

trap — Overflow triggers a runtime abort. Equivalent to Rust's debug builds. Safe but halts execution.

wrap — Overflow wraps around (two's complement). Identical to C's unsigned arithmetic and Rust's release builds. Fastest; behaviour is defined but may be unintended.

saturate — Overflow clamps to the type's maximum or minimum value. Common in signal processing and audio code.

default: trap

stack_size

Sets the initial thread stack size. Relevant for recursive algorithms, large stack frames, or embedded targets with constrained memory.

stack_size default  |  N (KB)

default — Platform default (typically 1–8 MB depending on OS).

N — Explicit size in kilobytes. Example: stack_size: 256 allocates a 256 KB stack.

default: default

safety

Controls whether the compiler inserts runtime safety checks. Does not affect formal verification — that is a separate compilation step controlled by the MemProof pipeline.

safety checked  |  unchecked

checked — Null pointer dereference, out-of-bounds access, and type mismatch checks are inserted at compile time. Violations abort with a diagnostic message.

unchecked — No runtime checks. Behaviour on violation is undefined. Use in production hot paths where you have verified safety separately.

default: checked

bounds_check

Specifically controls array and pointer bounds checking, independently of safety. Useful when you want all other safety checks but need tight loops to run without bounds overhead.

bounds_check enabled  |  disabled

enabled — Every array access and pointer offset is range-checked at runtime.

disabled — Bounds checks are omitted. The compiler will warn if this is set without safety: unchecked.

default: enabled

int_width

Sets the width of the unqualified int type. Does not affect explicitly-sized types (int32, int64, etc.).

int_width 32  |  64  |  platform

32 — int is always 32-bit signed, regardless of target architecture.

64 — int is always 64-bit signed.

platform — int matches the platform's native integer width. Equivalent to C's int; not recommended for portable code.

default: 32

float_model

Controls floating-point semantics.

float_model ieee  |  fast

ieee — Strict IEEE 754 compliance. Results are fully reproducible across platforms. Required for numerical software, financial code, and anything where floating-point output must be deterministic.

fast — Allows the compiler to reorder, fuse, and reassociate floating-point operations for performance. Equivalent to -ffast-math. Results may differ by small amounts across architectures.

default: ieee

types

Determines how type mismatches are handled.

types strict  |  dynamic

strict — All type coercions are explicit. Mixing int and float in an expression is a compile error without a cast.

dynamic — Common numeric promotions are inserted automatically (int → float, narrower → wider). Closer to how Python or JavaScript handle arithmetic.

default: strict

concurrency

Sets the concurrency model for the file. Affects how the standard library handles I/O, timers, and inter-task communication.

concurrency threads  |  async  |  cooperative

threads — OS threads. Preemptive scheduling. Standard library primitives map to pthreads/Win32 threads. On the Python target, maps to threading for I/O-bound work or multiprocessing for CPU-bound work — the transpiler selects the correct backend based on usage.

async — Asynchronous tasks with an event loop. Functions that block are marked with @await. Maps to JS Promises on the JavaScript target, asyncio on the Python target, and virtual threads on the JVM target.

cooperative — Green threads with explicit yield points. Scheduling is application-controlled with no OS involvement. Useful for embedded systems and game loops.

default: threads

error_model

Sets how runtime errors and exceptional conditions are reported and propagated.

error_model exceptions  |  result  |  abort

exceptions — try/catch/error syntax. Implemented via setjmp/longjmp — no heap allocation, no C++ runtime dependency.

result — Functions that can fail return a result type. Errors are values; callers must handle them explicitly. Similar to Rust's Result<T, E> or Go's multiple return values.

abort — Errors call abort() immediately. For environments where error recovery is not possible or needed.

default: exceptions

exceptions

Controls how exception declarations are handled when transpiling to targets with checked exception systems. Only relevant when error_model is exceptions.

exceptions unchecked  |  checked

unchecked — Exceptions are not declared in function signatures. On the JVM target, the transpiler wraps all exceptions in RuntimeException so callers are never forced to handle them. This is the standard behaviour for Pragma code and all other targets.

checked — The transpiler generates explicit throws declarations on the JVM target, propagating exception types up the call stack as Java requires. Use this when integrating with Java libraries that expect checked exceptions to be declared. Has no effect on non-JVM targets.

default: unchecked

debug

Controls the presence of debugging information in the compiled output.

debug symbols  |  stripped

symbols — DWARF/PDB debug symbols are emitted. Stack traces contain file names and line numbers.

stripped — No debug information in the binary. Use for production release builds.

default: symbols
Settings are declared in a comment block at the top of each source file. If a setting is omitted, the compiler uses the default listed above. Settings do not propagate across files — each file is compiled according to its own declarations.
