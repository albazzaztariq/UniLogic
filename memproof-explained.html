A Basic Explanation of a Mathematical Proof Methodology Applied to Source Code

This page describes what MemProof does at each stage — what goes in, what comes out, and roughly why. It is not a formal treatment. For that, see the formal design document.

1. Safe Subset Check

If the function or module is marked @safe, the compiler first checks that the code uses only a restricted set of constructs: no dynamic allocation, no unbounded recursion, all loops with visible bounds or annotations, no function pointers in safety regions. This is the same model Airbus uses for avionics code. Code that passes this check verifies at 90–95% automatic coverage. A violation is a compile error before MemProof even runs.

2. IR Construction

The UL AST is lowered into typed SSA (Static Single Assignment) form. In SSA, every variable is assigned exactly once — any reassignment creates a new version of the variable. The control flow is mapped as an explicit graph with loop structures and branch conditions identified precisely. This is the internal representation everything else in the pipeline operates on.

3. Abstract Interpretation

This step runs through the entire IR without executing it, tracking the possible values of every variable at every program point — across all possible inputs, branches, and loop iterations. A loop counter that starts at 0 and increments to n gets tracked as i ∈ [0, n]. A sensor input declared as 0–1000 PSI gets tracked as pressure ∈ [0, 1000]. For loops, a technique called widening forces convergence when bounds aren't immediately visible from the code; narrowing then recovers the precise bound. The result is a guaranteed over-approximation: every value a variable could possibly hold at any point is captured.

4. Separation Logic and Shape Analysis

While abstract interpretation tracks scalar values (integers, booleans), separation logic tracks the heap — ownership of memory regions, which pointers alias, what each allocation contains. Its key property is local reasoning: a function can be verified using only the memory it actually touches, without reasoning about the entire program heap at once. This is what makes the approach scale to real programs. Shape analysis extends this to complex structures like linked lists and trees using inductive definitions that can be reasoned about symbolically without unrolling the structure.

5. Verification Condition Generation

For every potentially unsafe operation in the program, the VCG generates a precise mathematical question — a verification condition. For an array access arr[i]: is it possible for i to be negative, or to be greater than or equal to the array's length at this point in the code? For a pointer dereference: is it possible for the pointer to be null or to point to freed memory? Each question is enriched with everything abstract interpretation learned: the known value ranges at that point, the heap ownership facts from separation logic. This context is what lets the next stage answer efficiently.

6. SMT Solving (CVC5)

The solver takes each verification condition and tries to find a counterexample — a concrete input that produces a violation. It applies theory solvers for linear integer arithmetic, bitvectors, arrays, and heap predicates. When a partial assignment is found to be impossible, conflict-driven learning rules out entire regions of the search space at once, rather than continuing to explore them. When the solver exhausts all possibilities and finds none that cause a violation, it returns UNSAT. This is a formal proof by refutation, not a heuristic — it has genuinely checked every case. If it finds a counterexample, it returns SAT with the specific input values that trigger the violation. That is your bug report.

7. Proof Witness

For each UNSAT result, CVC5 produces an LFSC proof witness — a formal mathematical object encoding the entire reasoning chain that led to the conclusion. This witness is checked by a small, separately verified kernel of approximately 5,000 lines of auditable code. The kernel has no heuristics and no complex algorithms — it mechanically confirms whether a proof term is valid or not. Even if CVC5 had a bug that produced a wrong UNSAT result, a malformed witness would be rejected by the kernel. The trusted base shrinks to the kernel, not the full solver.

8. Certificate and Publication

The proof witnesses for all verification conditions, bundled with metadata — program hash, verifier version (as an IPFS content ID), each VC's location and result, and the proof witness hashes — form the certificate. Proof witnesses and the MemProof Coq source go to IPFS and Arweave. The certificate JSON, roughly 1KB, goes to the blockchain. Anyone can retrieve the MemProof Coq source from IPFS, check the proof with coqc, and confirm the verifier itself is formally sound — not just that it produced a result for your program.
